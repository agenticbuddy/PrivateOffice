# PrivateOffice — все изменения редактора (Collabora Online)

**Что это.** Полный список того, что наш редактор меняет относительно оригинала (апстрима), и — главное —
объяснение *как это устроено*, чтобы человек **без предварительного контекста** мог это сопровождать,
чинить и перевести интерфейс **на новый язык**.

**Кому.** Новому инженеру/переводчику. Читать сверху вниз: сначала «Как читать» и «Глоссарий», потом разделы.

## Как читать этот документ (обозначения)

**Схемы идентификаторов** (в тексте встречаются три вида ссылок):
- `PO-XXX-NNN` (напр. `PO-THEME-001`) — **единица кастомизации**, запись в реестре `editor/manifests/patchset.json`.
  Префикс = тип: THEME (тема), BRAND (ребренд), ASSET (картинки/иконки), CODE (правка кода), CSP, URL, CACHE (кэш),
  SRC (патч исходника), L10N (переводы), QA. Номер — просто порядковый. Полное описание каждой единицы — в приложении и в самом `patchset.json`.
- `0001`…`0006` — файлы **патчей исходного кода** в каталоге `editor/patches/` (напр. `0001-notebookbar-localized-function-names.patch`).
  Нумерация = порядок применения; «пропусков» нет — просто столько файлов. НЕ путать с `PO-*`.
- `#NN` (напр. `#58`, `#66`) — **номер замечания пользователя** (баг-репорт из истории работы), для трассировки. Не файл и не тег в коде.

**Пути** даны от корня репозитория (напр. `editor/l10n/overrides/client/ru.json`). `<lang>` — плейсхолдер языка
(`ru`, `de`, `fr`…); `dist/` — каталог собранного редактора **внутри образа** (`/usr/share/coolwsd/browser/dist/`).

## Глоссарий (обязательно к прочтению)

| Термин | Что это |
|---|---|
| **Collabora Online (CO)** | Open-source онлайн-редактор офисных документов (веб-надстройка над LibreOffice). Наш редактор — это CO, кастомизированный под PrivateOffice. |
| **CODE** | Collabora Online **D**evelopment **E**dition — бесплатная готовая сборка CO; наш `base image`. |
| **Апстрим (upstream)** | Оригинальный проект Collabora, от которого мы отталкиваемся. «Апстрим не кладёт X» = в оригинале X отсутствует. |
| **Движок / ядро / core** | Сам **LibreOffice** внутри CO: рендерит документы и «серверные» диалоги. У нас берётся **готовым бинарём** из base image, **не** собирается из исходников. |
| **coolwsd** | Серверный процесс CO (websocket-демон). Отдаёт HTML-страницу редактора и связывает браузер с ядром. Настраивается флагами `--o:ключ=значение` (см. ниже). |
| **browser bundle** | Клиентская часть редактора — JS и CSS, которые грузятся в браузер: `bundle.js` (логика) и `bundle.css` (стили). **Собирается из исходников CO.** |
| **cool.html** | HTML-обёртка редактора, которую отдаёт coolwsd. Содержит ссылки на `bundle.js`/`bundle.css` и плейсхолдеры (см. «плейсхолдер»). |
| **WOPI** | Протокол, по которому редактор берёт и сохраняет файл у «хоста» (нашего backend'а). |
| **notebookbar / лента (ribbon)** | Верхняя панель вкладок с кнопками (Главная, Вставка, Формулы…), как в MS Office. |
| **боковая панель (sidebar)** | Правая панель свойств (Стиль, Символы, Формат чисел, «Table Style Options»…). |
| **автофильтр** | Выпадающее меню фильтра на заголовке таблицы Calc (Сортировать / Фильтр по цвету / чекбоксы значений). |
| **пикеры** | Мелкие выборщики: цвета, шрифта, размера — маленькие всплывающие панельки. |
| **тултип** | Всплывающая подсказка при наведении мыши. |
| **спарклайны** | Мини-графики внутри ячейки таблицы (термин Calc/Excel). |
| **условное форматирование** | Правила подсветки ячеек по значению (термин Calc/Excel). |
| **UNO** | API/компонентная модель LibreOffice. Команды (`.uno:Save` и т.п.) имеют текстовые **метки**, которые берутся из отдельного «UNO-каталога» переводов. |
| **`_()`** | Функция-обёртка в клиентском JS, помечающая строку как переводимую (аналог `gettext`). При рендере подставляет перевод из **клиентского каталога**. `«_()-строка»` = строка, которую переводит клиент. |
| **gettext** | Стандартная система переводов. `.po` — **текстовый редактируемый** файл перевода; компилируется утилитой `msgfmt` в `.mo` — **бинарный** файл, который читает ядро. |
| **`.ui`** | XML-файлы LibreOffice, описывающие диалоги/панели. Ядро их рендерит и берёт переводы из `.mo`. |
| **sc / sw / sd** | Модули LibreOffice: **sc** = Calc (таблицы), **sw** = Writer, **sd/si** = Impress. Поэтому файл переводов Calc называется `sc.po` / `sc.mo`. |
| **Клиентский каталог** | JSON-файлы переводов клиента внутри `dist/l10n/`: `ui-<lang>.json` (общие `_()`-строки) и `uno/<lang>.json` (UNO-метки). |
| **Путь рендера (render path)** | Каким путём строка попадает на экран — см. раздел 1. Определяет, в каком файле её переводить. |
| **from-source сборка** | Мы собираем `browser bundle` и `coolwsd` **из исходников** CO (`online.mirror @ cp-26.04.1-4`), а ядро берём готовым. Делает `editor/Dockerfile.online`. |
| **sed** | Unix-утилита потоковой замены текста. Мы ею правим **уже собранные** `bundle.js`/`cool.html` в Dockerfile. |
| **source-патч (browser-патч)** | Unified-diff (`editor/patches/NNNN-*.patch`), применяемый `git apply` к **исходникам** CO **до** сборки. |
| **`configure`** | Шаг сборки CO из исходников (генерирует Makefile). Патчи применяем **до** него. |
| **CSP (Content Security Policy)** | Политика браузера: что можно грузить/исполнять. Inline-скрипты она обычно блокирует → скрипты грузим внешними файлами. |
| **первый кадр (first paint)** | Первая отрисовка страницы браузером. Тему применяем **до** неё, чтобы не мелькал неправильный вид. |
| **build-time vs serve-time** | **build-time** — при сборке образа (шаги в Dockerfile). **serve-time** — когда coolwsd **отдаёт** страницу пользователю (тогда он заменяет плейсхолдеры). |
| **плейсхолдер** | Метка в `cool.html` вида `%PRODUCT_BRANDING_NAME%` или `<!--%BRANDING_CSS%-->`, которую coolwsd заменяет **при отдаче** (serve-time). |
| **«протекает» (leak)** | Разработческий жаргон: наружу (в UI/перевод) просачивается то, чего не должно — напр. сырой id элемента или непереведённый английский фрагмент. |
| **cache-bust, `?b=`, `BUILD_ID`, `ver_suffix`, ETag** | Механизм заставить браузер перезабрать изменённые файлы. `BUILD_ID` = короткий git-хеш текущего кода. Добавляем `?b=BUILD_ID` к URL ассетов и в `ver_suffix` (→ HTTP-заголовок `ETag`). Новый URL/ETag ⇒ браузер грузит свежий файл, а не старый из кэша. |
| **patchset.json** | `editor/manifests/patchset.json` — **реестр** всех единиц кастомизации (типизированные записи: id, тип, файлы, обоснование, откат). |

**Флаги `--o:ключ=значение`** — параметры конфигурации coolwsd. Задаются в `docker-compose.yml` у сервиса `editor`
в строке `extra_params=…`. Пример: `--o:help_url=` (пустой URL справки), `--o:product_name=PrivateOffice`.

## 0. Из чего собран редактор (пин версий)
- **Base image:** `collabora/code@sha256:75859…` = CODE **26.04.1-4** (полный digest — в `editor/manifests/upstream.json`). Отсюда берётся готовое **ядро** (LibreOffice).
- **Browser bundle + coolwsd** собираются **из исходников** `online.mirror @ cp-26.04.1-4` против этого ядра (делает `editor/Dockerfile.online`).
- **Инвариант версий:** версия исходников `online` (`cp-26.04.1-4`) должна **совпадать** с версией ядра base image (CODE `26.04.1-4`) — клиент собирается ПРОТИВ этого ядра. При обновлении меняются **обе** (в `upstream.json`).
- Резервная сборка: `editor/Dockerfile` — патчит готовый образ без пересборки из исходников («stopgap», на случай проблем с from-source). ⚠️ Он **не применяет source-патчи** (раздел 5), поэтому боевой путь — только `Dockerfile.online`.
- Языков интерфейса: **22** — это те, что поддерживает апстрим CODE; **не предел**: новый язык добавляется (см. раздел 1, «Практика»). Приложения: Writer, Calc, Impress.
- Точные версии — в `editor/manifests/upstream.json`. **При обновлении версии (re-pin)** все source-патчи нужно пересобрать под новые исходники (иначе `git apply` упадёт — это намеренный сигнал; см. раздел 5 и `editor/patches/README.md`).

## 1. Локализация (переводы)

### Главный принцип: путь рендера
Одна и та же строка может попадать на экран **тремя разными путями**, и переводится она **в том файле,
который соответствует её пути** (иначе перевод лежит в каталоге, но экран остаётся английским):

1. **Клиент** — клиентский JS оборачивает строку в `_()` и подставляет перевод из `ui-<lang>.json`.
2. **UNO-метка** — если строка это метка команды, она берётся из `uno/<lang>.json`.
3. **Сервер/ядро** — если строку рисует ядро из `.ui`-файла (диалоги, боковые панели), перевод берётся из `.mo` (скомпилированного из `.po`).

**Как понять, какой путь у конкретной строки** (практически): найти английский оригинал в `dist/l10n/ui-<lang>.json`
и в `dist/l10n/uno/<lang>.json`; если есть там и рендерится по-русски — путь клиентский/UNO. Если строки нет в
клиентских каталогах, а на экране английский — почти всегда путь серверный (`.ui`→`.mo`). Финальная проверка —
**только рендер** (открыть элемент в живом редакторе).

### Где что переводится (слои)
Все наши переводы лежат данными в `editor/l10n/` и **вмерживаются в каталоги при сборке образа**
(шаги в `editor/Dockerfile.online`). «→» = «при сборке попадает в такой-то файл внутри образа».

| Путь рендера | Наш файл (правим здесь) | Куда попадает при сборке | Что покрывает | Объём для `ru` |
|---|---|---|---|---|
| Клиент `_()` | `editor/l10n/overrides/client/<lang>.json` | `dist/l10n/ui-<lang>.json` (наши ключи дописываются к каталогу) | лента, меню, тултипы, имена фигур/команд/диаграмм, отслеживание изменений, спарклайны, AI-строки, диалог импорта CSV | **1287** пар «англ→рус» |
| UNO-метка | `editor/l10n/overrides/uno/<lang>.json` | `dist/l10n/uno/<lang>.json` | метки команд (напр. «Таблица», «Вставить таблицу») | 2 |
| Сервер (Calc) | `editor/l10n/overrides/core/<lang>/sc.po` | компилируется `msgfmt` и **дописывается в** `.../resource/<lang>/LC_MESSAGES/sc.mo` ядра | серверные строки Calc: диалог «Создать таблицу»; боковая панель «Table Style Options» + метка «Категория»; вкладка «Проверка данных → Действие при ошибке» | 23 записи |
| Справка клавиш | `editor/l10n/help/<lang>.json` → готовый `editor/cool-help.ru.html` | копируется в `dist/cool-help.html` | весь справочник горячих клавиш (открывается кнопкой «Комбинации клавиш») | 527 строк |

Пояснения к таблице:
- **Счётчики** называются по-разному, потому что это разные форматы: «пары/ключи» — записи JSON; «записи» —
  блоки `msgctxt/msgid/msgstr` в `.po`; «строки» — переводимые текстовые узлы в HTML-справке. Все — просто «сколько переведено».
- **`sc.po`/`sc.mo`** — только модуль **Calc** (`sc`). Для Writer/Impress аналогов у нас пока нет (там серверных
  пробелов не нашли). При необходимости заводится `sw.po`/`sd.po` тем же способом.
- **Почему часть путей на `ru`, а не `<lang>`.** Полностью (client+uno+core+help) сейчас доведён только русский.
  Для остальных языков есть небольшие client-оверрайды (`ar` 11, `ja` 8, `ko` 7, `pt_BR` 8, `tr` 6, по 1 в
  `cs/fr/he/it/uk/vi`) — точечные заплатки к тому, что уже есть в апстрим-каталогах. Это НЕ ограничение движка,
  а объём проделанной работы.

### Патчи, которые чинят сам путь рендера
Иногда апстрим строит строку в коде так, что перевод не подхватывается. Тогда правится **исходник** CO
(файлы `editor/patches/*.patch`, применяются `git apply` при сборке), а данные к ним лежат в head-скриптах `editor/po-*.js`:

| Патч (файл) | Правит исходник CO (путь внутри дерева CO) | Что чинит | Данные |
|---|---|---|---|
| `0001-notebookbar-localized-function-names.patch` | `browser/src/control/jsdialog/Definitions.Menu.ts` | имена функций в дропдаунах ленты «Формулы» — И отображение, И **вставку** (ядро принимает только локализованное имя функции) | `editor/po-funcnames.js` — генерится `scripts/build-l10n-funcnames.py` из `.mo` ядра |
| `0004-localized-cell-style-names.patch` | `browser/src/control/jsdialog/Widget.Listbox.ts` | имена стилей ячеек (боковая панель «Стили» + дропдаун условного форматирования) | `editor/po-stylenames.js` — генерится `scripts/build-l10n-stylenames.py` |
| `0005-localized-theme-names.patch` | `browser/src/control/jsdialog/Widget.IconView.ts` | имена тем документа (галерея «Темы» на вкладке «Формат») | `editor/po-themenames.js` — **правится вручную** (движок не даёт этих переводов) |

Head-скрипты `po-*.js` подключаются в `cool.html` и грузятся ДО `bundle.js`, кладя данные в глобальные переменные,
которые читают патчи.

### Итог и как он проверен
Видимый интерфейс = **0 непереведённых `_()`-строк**. Английскими остаются только международные обозначения
форматов бумаги/файлов (`A4`, `B5 (ISO)`, `Letter`, `Legal`, `Markdown (.md)`) — их не переводят.

**Метод проверки полноты** (воспроизводимо): извлечь все строки, обёрнутые в `_( "…" )`, из отданного
`dist/l10n/…`/`bundle.js`, вычесть ключи из `ui-<lang>.json` — остаток должен быть пуст (кроме имён-форматов).
Плюс визуальная проверка рендером в живом редакторе (Playwright) — единственная финальная истина.

### 🔧 Практика: как добавить/поправить перевод или новый язык
0. **Откуда взять список английских оригиналов** (что вообще нужно перевести): все клиентские `_()`-строки —
   извлечь из отданного `bundle.js` (см. команду в разделе 8) или взять `dist/l10n/ui-en.json` как референс; серверные — из `.ui`-файлов движка (ниже).
1. **Найти путь рендера** строки (см. выше).
2. **Клиентская строка** → добавить пару `"English": "Перевод"` в `editor/l10n/overrides/client/<lang>.json`.
3. **UNO-метка** → в `editor/l10n/overrides/uno/<lang>.json`.
4. **Серверная (Calc)** → добавить запись в `editor/l10n/overrides/core/<lang>/sc.po`. **Точный `msgctxt`** ищется в `.ui`-файле движка внутри контейнера:
   `docker compose exec editor grep -rn 'context=' /opt/collaboraoffice/share/config/soffice.cfg/modules/scalc/ui/` (там же виден английский `msgid`). Формат — стандартный gettext.
5. **Справка клавиш** → перевести строки в `editor/l10n/help/<lang>.json`, затем **перегенерировать** `cool-help.<lang>.html`: взять штатный `dist/cool-help.html` и заменить его текстовые узлы переводами из `help/<lang>.json` (скриптом-подстановкой), результат закоммитить и добавить `COPY … dist/cool-help.html` в `Dockerfile.online`.
6. **Новый язык (напр. немецкий `de`):** как минимум создать `editor/l10n/overrides/client/de.json`; для полного покрытия — так же `uno/de.json`, `core/de/sc.po`, `help/de.json`. Пересобрать: `./scripts/up.sh editor`. Проверить рендером.
7. **Пересборка** после любой правки в `editor/**`: `./scripts/up.sh editor` (пересобирает редактор и проставляет свежий cache-bust-токен — см. раздел 6).

## 2. Стили / дизайн (тема)

- Все наши стили — в одном файле `editor/branding-privateoffice.css` (~1478 строк). При сборке образа он
  **дописывается в конец** `bundle.css` (штатного клиентского CSS) — командой `cat` в `editor/Dockerfile.online`
  (единица `PO-THEME-001`). Так тема — часть собственного стиля редактора, а не «наложенный сверху» override-файл.
- **Переключение дизайна** делает скрипт `editor/po-toggle.js` (грузится в `<head>` `cool.html` до отрисовки).
  Он читает параметр URL `?po_design` и выставляет на теге `<html>` атрибут `data-po`. Значения:
  `?po_design=glass` → `data-po="glass"`; `?po_design=glass2` → `data-po="glass2"`; `?po_design=classic` (или без параметра) → атрибут не ставится.
  Правила в CSS привязаны к этому атрибуту (`html[data-po="glass2"] …`), поэтому нужный дизайн активен уже на **первом кадре** (не мелькает), и это не нарушает **CSP** (скрипт внешний, не inline).
- **Три дизайна:** `classic` (стоковый вид CO, наших правил нет) · `glass` (зелёный акцент, ~219 CSS-правил, привязанных к `data-po="glass"`) · `glass2` (синий акцент `#2563d9`, ~223 правила). Все три — в том же одном файле, различаются только префиксом-селектором.
- **Что охвачено темой:** лента, меню, всплывающие окна (попапы), диалоги, боковая панель, пикеры, тултипы, автофильтр.
- **Точечные стиль-фиксы** (накоплены в истории; все — в `editor/branding-privateoffice.css`):
  - **комбобокс = одно поле:** у выпадающих полей ввода убраны двойная рамка и слишком узкая рамка-обводка при фокусе (было визуально «поле в поле»).
  - **accent-пилюли дропдаунов ленты:** пункты выпадающих меню ленты (напр. под кнопкой «Автосумма ▾») получили скруглённый фон акцентного цвета при наведении/выборе («пилюля» = скруглённый прямоугольник).
  - **тумблеры боковой панели:** выбранное состояние переключателей в сайдбаре показывается контуром (outline) акцентного цвета, а не сплошной заливкой.
  - **переключатель режима «Просмотр/Редактирование»:** кнопка была серой/невыразительной → перекрашена в акцентный цвет.
  - **фикс наложения попапов:** со всей ленты убрано CSS-свойство `backdrop-filter` (размытие фона) — оно ломало наложение всплывающих окон поверх ленты.
  - **автофильтр (#58):** выделение строк в меню автофильтра сделано мягким акцентным (светлый фон + тёмный акцентный текст + тонкая рамка) вместо тяжёлой сплошной синей заливки — читается легче и единообразно с лентой/сайдбаром.

## 3. Ребрендинг (Collabora → PrivateOffice)
Имя продукта задаётся в **нескольких местах**, потому что оно всплывает на разных этапах; все они приводятся к «PrivateOffice»:

- **`PO-BRAND-003`** — в **собранных** файлах (`dist/**/*.{js,html,json,css}`) заменяем строки «Collabora …» на «PrivateOffice» утилитой `sed`.
  Замена по **границе слова** (word boundary в регулярке) — чтобы не испортить слова вроде «Collaborative». Это касается **отображаемых** строк и текста копирайта.
- **`PO-ASSET-008`** — картинки логотипа заменяются нашим `editor/po-logo.svg`: это касается **вордмарка** (текстовый логотип) в окне «О программе» и логотипа в **верхней панели** редактора.
- **`PO-BRAND-011`** — coolwsd подставляет **при отдаче страницы** (serve-time) плейсхолдеры `%PRODUCT_BRANDING_NAME%` / `%PRODUCT_BRANDING_URL%` — их `sed` из п.1 (он работает при сборке) не видит. Поэтому прямо в `cool.html` задаём их значения (PrivateOffice / `about:blank`), очищаем feedback/welcome/buyproduct и удаляем экран приветствия (welcome). `about:blank` = пустая страница браузера: так ссылка «о продукте» никуда не ведёт (не на сайт Collabora).
- **Флаг** `--o:product_name=PrivateOffice` в `docker-compose.yml` — задаёт имя продукта на уровне сервера coolwsd.

(Замена «Collabora → PrivateOffice» — по отображаемым строкам во всех файлах, независимо от языка.)

## 4. Приватность / «no phone-home»
«**Phone-home**» = когда программа сама, без ведома пользователя, обращается к серверам производителя
(проверка обновлений, телеметрия, внешние ссылки). Мы это убираем:

- **`PO-URL-007`** — все апстрим-URL (ведущие на ресурсы Collabora) в собранных файлах заменяем на `about:blank`:
  ссылка под логотипом в шапке, «release notes» (что нового), «docs» (документация), «proxy» (прокси-настройки). У логотипа есть URL, потому что апстрим делает его кликабельным на свой сайт.
- **`--o:help_url=`** (пустой) в `docker-compose.yml` — coolwsd тогда **скрывает все кнопки внешней справки**.
  А патч **`0006-remove-collabora-external-links.patch`** (правит `browser/src/canvas/sections/FormulaErrorHelpSection.ts` в исходнике CO) убирает оставшиеся ссылки на сайт Collabora: Forum, Report an issue, Latest Updates, Send Feedback и «help по ошибке формулы».
- **`--o:fetch_update_check=0`** и **`--o:allow_update_popup=false`** — отключают проверку обновлений и попап о новой версии («Latest Updates» из п. выше — это ссылка в справке; эти флаги — про фоновую проверку; они независимы, но оба про «не звонить домой»).

Разграничение: имя/URL/логотип задаются **флагами и заменами строк**; удаление конкретных ссылок в UI — **патчем кода** (0006), потому что там ссылки зашиты в исходник.

## 5. Код-фиксы (функциональные баги)
Здесь чинится **поведение**, не текст. Два способа: **source-патч** (правка исходника CO до сборки) и **post-build sed** (замена в уже собранном `bundle.js`).

**Source-патчи** (`editor/patches/*.patch`, применяются `git apply` до `configure`):
- **`0002-treeview-clear-selection.patch`** (`browser/src/control/jsdialog/Widget.TreeView.ts`) — когда приходит «выделена строка −1», трактовать это как «снять выделение», а не как ошибку (убирает спам `console.warn` в консоли).
- **`0003-notebookbar-keep-formulas-tab.patch`** (`browser/src/control/Control.Notebookbar.js`) — после вставки функции вкладка «Формулы» остаётся активной (заодно исправлена опечатка во **внутреннем имени вкладки** в коде — из-за неё вкладка сбрасывалась).

**Post-build sed** (команды в `editor/Dockerfile.online`, правят собранный `dist/bundle.js`):
- **`PO-CODE-004`** — строку «More options for {1}» брать из **каталога переводов** (клиентского `ui-<lang>.json`), а не собирать сырым склеиванием `'More options for '+id` (иначе в UI **протекал** внутренний id элемента, и строка не переводилась).
- **`PO-CODE-005`** — то же для «Open {name}» (апстрим склеивал непереводимые фрагменты → английский протекал, порядок слов ломался).
- **`PO-CODE-010`** — в функции `getImageURL` при пустом имени иконки возвращать `''` (иначе запрашивался `images/` → ошибка 404).
- **`PO-ASSET-009`** — досоздать файл иконки `dist/images/lc_pastespecial.svg` (светлую и тёмную): ядро её запрашивает, а апстрим не кладёт → 404 и пустая иконка. Копируем из имеющейся иконки вставки.
- **`#61`** — тултип «Последнее сохранение: N мин. назад» вместо «-36 мин». Причина: JS-функция `Intl.RelativeTimeFormat` со стилем `narrow` для русского даёт «знаковое число» («-36 мин»). `sed` меняет стиль `narrow`→`short` → получается нормальное «36 мин. назад».
- **`PO-CSP-002`** — удалён мёртвый inline-`<script>` тёмной темы из `cool.html`: он нарушал CSP на каждой загрузке и всё равно никогда не исполнялся.

Проверка код-фиксов — рендером в живом редакторе (Playwright); для стилей/модалок, которые не попадают в скриншот, — чтением вычисленных стилей (`getComputedStyle`).

## 6. Кэш / сборка (чтобы браузер не отдавал старое)
Проблема: браузер кэширует `bundle.js`/`bundle.css` и после нашего изменения может показывать **старую** версию.
Решение — **cache-bust**: у каждой сборки свой маркер, который меняет URL/ETag, и браузер вынужден перезабрать файл.

- `./scripts/up.sh` (наш скрипт запуска) вычисляет **`BUILD_ID`** = короткий git-хеш текущего кода (или `<хеш>-dirty`, если есть незакоммиченные правки).
- **`PO-CACHE-012`** — прописать `BUILD_ID` в `ver_suffix` в `coolwsd.xml` → это меняет HTTP-заголовок **`ETag`** ассетов (ETag = идентификатор версии файла для кэша браузера).
- **`PO-CACHE-013`** — дописать `?b=BUILD_ID` ко всем ссылкам на `*.css/js/json` в `cool.html`. Новый URL ⇒ браузер грузит свежий файл.
- **`PO-BRAND-011`** — свой **пустой (инертный)** `branding.js`, подключаемый **нашим** тегом в `cool.html`, чтобы он тоже попал под `?b=`. Если этого не сделать, coolwsd подставит `%BRANDING_JS%` **при отдаче** своим тегом **без** `?b=` → браузер отдаст старый закэшированный бренд-скрипт.
- **`#66`** — то же для `branding.css`: подключаем его **своим** `<link>` вместо serve-time плейсхолдера `%BRANDING_CSS%`, чтобы он тоже был под `?b=`. **Зачем:** устаревший закэшированный `branding.css` нёс правило `.jsdialog-window{overflow:hidden}`, которое обрезало (клипало) диалоги — из-за этого чекбоксы в диалогах не нажимались. Cache-bust это устранил.

Итог: правило проекта — **`?b=`-токен отданных ассетов должен совпадать с git HEAD**; иначе браузер отдаёт старое.

## 7. Как это собирается (конвейер `editor/Dockerfile.online`, по порядку)
Боевой образ собирается **из исходников**. Порядок шагов (важен):
1. `git clone` исходников CO (`online.mirror @ cp-26.04.1-4`).
2. **`git apply`** каждого `editor/patches/*.patch` к клонированному дереву (единица `PO-SRC-014`). Патч, который не применился, **валит сборку намеренно** — это сигнал, что при обновлении версии его надо пересобрать (rebase).
3. `configure` + сборка `browser bundle` и `coolwsd`.
4. **Мерж переводов:** дописать наши `client/<lang>.json` и `uno/<lang>.json` в собранные каталоги `ui-<lang>.json` / `uno/<lang>.json`.
5. **Core-переводы:** скомпилировать `core/<lang>/sc.po` через `msgfmt` и дописать в `sc.mo` ядра — **до** «systemplate-setup» (подготовки шаблона документа), иначе не попадёт.
6. **Замены `sed`** по собранным файлам: ребренд (раздел 3), нейтрализация URL (раздел 4), код-фиксы (раздел 5).
7. Замена логотипов на `po-logo.svg`; досоздание недостающих иконок.
8. `COPY editor/cool-help.ru.html` → `dist/cool-help.html` (русская справка клавиш).
9. **Cache-bust:** проставить `ver_suffix`/`ETag` и `?b=BUILD_ID` (раздел 6).

Каждая единица этого конвейера зафиксирована как типизированная запись в `editor/manifests/patchset.json`
(`id`, `type`, `files`, обоснование, как откатить).

## 8. Как проверять сделанное
- **Рендер (Playwright)** — единственная финальная истина для UI. Тесты в `e2e/` (`cd e2e && npm test`).
- **Стили модалок/меню** — они НЕ попадают в скриншот (рисуются отдельным слоем). Проверять **чтением
  вычисленного стиля** на живом элементе, а не по картинке. Идея: в консоли iframe редактора
  `getComputedStyle(document.querySelector('<селектор элемента>'))` и сравнить нужное свойство (цвет, фон и т.п.).
- **Полнота перевода** (для языка `<lang>`) — извлечь все строки в `_("…")` из отданного `bundle.js` и вычесть
  ключи каталога `ui-<lang>.json`; остаток должен быть пуст, **кроме имён-форматов** (международные обозначения
  бумаги/файлов — `A4`, `B5 (ISO)`, `Letter`, `Legal`, `Markdown (.md)` — их не переводят). Черновая команда:
  ```bash
  H=$(git rev-parse --short HEAD)
  curl -s "http://localhost:8088/browser/<hash>/bundle.js?b=$H"          > /tmp/b.js
  curl -s "http://localhost:8088/browser/<hash>/l10n/ui-ru.json?b=$H"    > /tmp/ui.json
  # затем скриптом: множество _("…") из b.js  МИНУС  ключи ui.json  → должно быть пусто (кроме имён-форматов)
  ```
  (`<hash>` — путь версии в URL ассетов, виден в `cool.html`.) То же покрытие мы проверяли по UNO/core — но для
  них «истина» именно рендер (клиентский extract их не видит).
- **Cache-bust** — `?b=<токен>` у отданных ассетов должен совпадать с git HEAD:
  ```bash
  docker compose exec editor grep -oE 'bundle\.css\?b=[a-z0-9]+' \
    /usr/share/coolwsd/browser/dist/cool.html   # → сравнить с  git rev-parse --short HEAD
  ```
  Если видите токен вида `<хеш>-dirty` — есть незакоммиченные правки; **закоммитьте и пересоберите**
  (`./scripts/up.sh editor`), иначе браузер может отдавать старый кэш.
