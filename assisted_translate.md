# Ассистированная локализация Collabora Online

Этот документ описывает осторожный рабочий процесс для полной локализации проекта на базе Collabora Online с помощью ИИ-агента. Он не привязан к конкретному инструменту. Под "агентом" ниже понимается любой помощник, который умеет читать проект, запускать команды, анализировать код, готовить отчеты, предлагать правки и проверять результат: Codex CLI, Claude Code, OpenRouter-агент, локальный LLM-агент или похожий инструмент.

Главная идея: локализация Collabora Online не сводится к заполнению пропущенных ключей в JSON. Надо сначала понять, где именно каждая английская строка используется, каким механизмом она переводится, есть ли у нее разные контексты, и только потом делать точечные правки.

Документ специально написан как рабочая инструкция, а не как краткая памятка. Если инженер впервые видит Collabora Online, важно не только знать список файлов, но и понять, почему простые действия вроде "добавить перевод в JSON" могут дать ложноположительный результат: один экран исправится, другой сломается, а проверка по grep покажет, что "перевод есть".

Отдельный акцент сделан на контроле ИИ-агента. Агент может уверенно сказать, что "проверил все языки" или "построил полную таблицу", но на практике часто делает выборку: несколько языков, пару вкладок, первые совпадения grep, часть каталогов. Поэтому результат работы агента надо принимать только тогда, когда он оставил проверяемые артефакты: машинную таблицу, manifest сценариев, список входных файлов с hash, скриншоты, DOM/text dumps, команды воспроизведения и числа, которые можно пересчитать независимо.

## Короткий вывод

Если строка не переведена в `ui-*.json`, это еще не значит, что можно просто добавить перевод и считать задачу закрытой.

Причины:

- Один английский `msgid` может встречаться в нескольких местах интерфейса.
- В разных местах один и тот же английский текст может требовать разных переводов.
- Некоторые строки являются частями составных фраз, а не самостоятельными фразами.
- Некоторые подписи приходят не из `ui-*.json`, а из UNO или LibreOffice-core.
- Некоторые английские значения лучше оставить английскими: названия форматов, технические термины, имена функций, бренды.
- Перевод может быть правильным лингвистически, но сломать ширину кнопки, RTL-раскладку, tooltip, меню или диалог.
- Перевод может уже лежать в каталоге, но все равно не появляться на экране, если строку рисует другой слой.
- Готовые чужие переводы (свежий языковой пакет LibreOffice) нельзя подкладывать вслепую: Collabora отличается от стокового LibreOffice, часть строк не совпадет.

Хорошая новость при этом: **почти вся локализация и все тексты, размеры и дизайн элементов интерфейса достижимы без сборки ядра LibreOffice** — они живут в слое Collabora Online (данные + CSS + немного кода). Сборка ядра нужна только для отрисовки самого документа.

Поэтому правильный путь такой:

1. Собрать все английские строки.
2. Собрать все места, где каждая строка встречается.
3. Разделить строки по источнику: `ui-json`, `uno-json`, `lo-core-mo`, frontend i18n, generated/composed text, unresolved.
4. Построить карту контекстов и возможных коллизий.
5. Переводить сначала безопасные случаи.
6. Для рискованных случаев делать контекстное разделение или править правильный PO/MO/UNO-источник.
7. Проверять каждую правку рендером, а не только grep-ом по файлам.

## Что именно сложно

В обычном приложении часто есть один слой локализации: код вызывает `t("Save")`, каталог хранит перевод, интерфейс показывает перевод.

В Collabora Online слоев больше:

| Слой | Что в нем живет | Как обычно чинить |
| --- | --- | --- |
| Frontend приложения | Обертка проекта вокруг Collabora: логин, файлы, профиль, модалки | `frontend/src/i18n/messages/*.json` |
| Collabora browser UI | Ribbon/notebookbar, часть меню, tooltips, панели, кнопки в web-коде | `l10n/ui-<locale>.json` — данные в `editor/l10n/overrides/client/<lang>.json`, их мёрджит deployed-сборка `editor/Dockerfile.online` (`editor/Dockerfile` — stopgap) |
| UNO command labels | Подписи команд LibreOffice, которые Collabora вызывает через `_UNO(...)` | `l10n/uno/<locale>.json` или источник генерации этих каталогов |
| LibreOffice-core | Диалоги, server/core UI, `.ui`, gettext, некоторые sidebar и dialog строки | `.po` -> `.mo`, затем пересборка или замена ресурсов в образе |
| Составные строки | Фразы, собранные из нескольких переводимых или непереводимых кусков | Переписать на цельную фразу с placeholders |
| Системные элементы | Браузерные меню, OS-native элементы, некоторые file picker/clipboard prompts | Обычно вне контроля приложения |

Важно: если строка относится к `ui-json`, правка `.mo` может ничего не дать. Если строка относится к `lo-core-mo`, добавление ключа в `ui-ru.json` может ничего не дать. Поэтому сначала надо определить источник.

И даже принадлежность к источнику нельзя выводить из того, что строка просто лежит в каком-то каталоге. Пример: строка `Clipboard` есть в `ui-ru.json` с русским переводом, но в ленте она все равно показывается по-английски - потому что эту подпись рисует ядро, а клиентский каталог в этом месте не используется. То есть "строка есть в каталоге" и "строка реально берется из этого каталога" - разные вещи. Настоящий источник подтверждается только экспериментом: поменять строку в каталоге-кандидате, пересобрать, сбросить кэш и посмотреть, изменился ли экран.

Отдельно надо понимать, что одна и та же английская строка может одновременно лежать в нескольких слоях - и в клиентском каталоге, и в `.mo` ядра. Поэтому источник строки не обязательно одно значение из списка; это набор слоев, где она встречается. Какой именно слой реально рисует текст на экране, решает render path (см. ниже), а не сам факт наличия строки в каком-то из каталогов.

На практике, когда это измеряли на реальном проекте, большинство пропусков оказалось в клиентских JSON, а не в ядре; при этом часть строк, формально присутствующих в каталоге, все равно рисовалась по-английски. Поэтому раскладку "что чем чинить" нельзя брать на веру - ее надо проверять на экране.

## Почему прямое добавление перевода опасно

Представим ключ `Illustrations`.

Если он используется только как название группы ribbon, перевод `Иллюстрации` может быть корректным.

Но если где-то есть составное сообщение вида:

```text
No + Illustrations
```

то простой словарный перевод даст что-то вроде:

```text
Нет Иллюстрации
```

а правильно по-русски:

```text
Нет иллюстраций
```

Для английского такая нарезка часто выглядит нормально, потому что английский терпит короткие словарные куски. Для русского, украинского, арабского, иврита, фарси, японского, корейского, тайского и многих других языков такая нарезка может быть неправильной.

Еще примеры рискованных слов:

| Английская строка | Почему рискованно |
| --- | --- |
| `Line` | Линия как фигура, строка текста, граница, линия диаграммы, команда меню |
| `Table` | Таблица как объект, команда вставки, стиль, вкладка, группа |
| `Set` | Глагол "задать", существительное "набор", технический "set" |
| `Field` | Поле формы, поле данных, поле сводной таблицы |
| `Function` | Функция Calc, команда вставки функции, раздел библиотеки функций |
| `Range` | Диапазон ячеек, область, именованный диапазон |
| `General` | Общий формат числа, общий раздел настроек, обычный режим |
| `Delete` | Удалить объект, удалить строку, удалить лист, удалить комментарий |

Вывод: перевод должен быть привязан не только к английскому тексту, но и к месту употребления.

## Базовые термины

| Термин | Значение |
| --- | --- |
| `msgid` | Английская исходная строка или ключ, по которому ищется перевод |
| occurrence | Конкретное место, где строка используется в коде или рендере |
| context | Окружение строки: приложение, вкладка, группа, команда, tooltip, соседние слова |
| collision | Ситуация, когда один `msgid` используется в разных смыслах |
| composed fragment | Строка является частью фразы, а не самостоятельной фразой |
| source class | Слой (или слои) локализации, где живет строка: `ui-json`, `uno-json`, `lo-core-mo`, frontend i18n и так далее. Может быть несколько сразу; какой из них реально рисует текст, определяет render path |
| baseline | Зафиксированное состояние до правки: версия образа, каталоги, скриншоты, списки строк |
| target locale | Язык, который надо проверять или допереводить |
| rendered proof | Доказательство через браузер: скриншот, DOM/text inventory, состояние UI |
| render path | Чем строка реально отрисована на экране: клиентский `_()`, серверный JSDialog из ядра или UNO. Определяется экспериментом, а не наличием в каталоге |
| plural form | Форма слова, зависящая от числа (1 файл / 2 файла / 5 файлов). В русском 3 формы, в арабском - 6 |

## Общий рабочий процесс

Работу лучше делить на этапы. Нельзя начинать с массовой правки переводов.

## Принцип доказательности

Самое надежное доказательство здесь - это то, что реально нарисовано на экране. Скриншот - единственный ground truth. JSON-каталог, grep, содержимое `.mo` и даже сам факт вызова `_("...")` в коде - это только гипотезы: они говорят, что перевод *может* примениться, но не доказывают, что он применился именно в этом месте. Поэтому источник и перевод каждой строки подтверждаются отдельным проходом **по каждому объекту** интерфейса (не по выборке) - открыть элемент и посмотреть глазами. В идеале на каждый блок (вкладка, группа, кнопка, меню, контекстное меню, диалог, всплывающая подсказка) делается скриншот. Затем отдельный, второй проход - желательно другой, более сильной моделью - просматривает эти скриншоты и ищет ошибки: первый проход собирает, второй проверяет.

Любой этап должен заканчиваться не фразой "готово", а набором артефактов, по которым другой человек или другой агент может пересчитать результат.

Плохой результат:

```text
Я проверил все строки и нашел 20 проблем.
```

Почему плохо:

- неясно, какие файлы проверялись;
- неясно, какие языки были в scope;
- неясно, где остальные строки;
- нельзя отличить полный проход от выборки;
- нельзя повторить результат после обновления образа.

Хороший результат:

```text
Входные файлы:
- bundle.js sha256=...
- ui-ru.json sha256=...
- uno/ru.json sha256=...

Scope:
- apps: Writer, Calc, Impress, shared builder
- locales: 22, список приложен
- excluded: browser-native dialogs, OS file picker

Артефакты:
- localization_occurrences.json: 1842 occurrence
- localization_coverage.csv: 22 locales x 436 direct keys
- localization_unresolved.csv: 54 commands
- screenshots/: 22 locales x 3 apps x 12 states

Проверочные числа:
- direct _() matches in extracted segments: 436 unique msgid
- _UNO commands: 264 unique command id
- global UNO labels: 813 unique msgid
```

Хороший результат можно перепроверить:

- открыть входной файл по hash и убедиться, что анализировалась правильная версия;
- пересчитать количество строк другим скриптом;
- проверить, что количество строк в отчете совпадает с количеством строк в JSON/CSV;
- выбрать любой `msgid` и найти все его occurrence;
- выбрать любой скриншот и сопоставить его с DOM/text dump;
- увидеть, какие места не покрыты.

### Что нельзя считать доказательством

Нельзя принимать как доказательство:

- общий текстовый ответ без raw-таблиц;
- скриншоты только для одного языка, если заявлено "все языки";
- grep-вывод без списка файлов и правил извлечения;
- отчет "нашел все", где нет общего количества найденных сущностей;
- таблицу "top missing strings", если нужна полная таблица;
- ручные выводы без machine-readable артефакта;
- перевод, для которого не указаны все occurrence;
- патч, для которого не указано, почему выбран именно этот source class;
- Playwright-проверку без manifest сценариев;
- отсутствие ошибок как доказательство полноты;
- наличие строки/перевода в каталоге (`ui/uno/locore-*.json` или `.mo`) как доказательство того, что перевод реально виден на экране (доказано обратное: `Clipboard` есть в `ui-ru.json`, но рисуется по-английски);
- render-проверку без cache-bust/hard-reload (кэш-ловушка: ассеты кэшируются под неизменным хэшем, и старый экран можно принять за результат фикса в любую сторону);
- один сброс кэша как гарантию свежести: часть ассетов (скрипт брендирования, рантайм-каталоги `ui-<loc>.json`) минует обычный cache-bust, поэтому рендер проверяют в чистом профиле браузера и дополнительно сверяют запросом мимо кэша.

### Минимальный набор проверяемых артефактов

| Этап | Что должен оставить агент | Чем это проверять |
| --- | --- | --- |
| Baseline | `baseline_manifest.json` с языками, приложениями, файлами, hash, версиями контейнеров | Пересчитать hash, сравнить языки с кодом проекта, проверить наличие файлов |
| Static extraction | `occurrences.json`, `coverage.csv`, `unresolved.csv`, команда запуска extractor | Перезапустить extractor, пересчитать regex/AST matches, проверить случайные строки вручную |
| Rendered inventory | `scenario_manifest.json`, screenshots, DOM/text dumps, trace/log | Проверить, что все пары locale/app/state присутствуют, открыть выборочные скриншоты |
| Collision map | `collisions.csv` со всеми `msgid`, risk class и occurrence ids | Проверить, что каждый `msgid` из occurrence map классифицирован ровно один раз |
| Translation candidates | `translation_candidates.csv` с source, context, confidence, reason | Проверить, что туда не попали hard cases и unresolved |
| Patch | diff, список approved candidate ids, список измененных keys | Проверить, что diff меняет только разрешенные source classes |
| Verification | before/after inventory, screenshots, failures, false positives | Сравнить изменения с patch list и проверить отсутствие новых конфликтов |

### Инварианты полноты

Для каждого этапа нужны числа, которые должны сходиться.

Примеры инвариантов:

- число языков в отчете равно числу языков в `backend/app/services/locales.py`;
- для каждого языка есть строка в coverage table;
- каждый `msgid` из occurrence map имеет хотя бы один occurrence;
- каждый occurrence ссылается на существующий файл, offset, selector или screen state;
- каждый rendered English leftover связан с occurrence map или помечен `unresolved`;
- каждый `unresolved` имеет причину и следующий шаг;
- каждый safe candidate имеет risk class `single-use` или `same-context multi-use`;
- каждый измененный перевод имеет approved candidate id;
- каждый patch key после сборки реально присутствует в активном контейнере;
- каждый измененный UI-state имеет before/after screenshot или text dump;
- каждый hard case либо имеет human decision, либо явно остается нерешенным.

Если инварианты не сходятся, работа не завершена. Даже если агент написал связный отчет.

### Этап 1. Зафиксировать baseline

Нужно понять, что именно мы локализуем.

Минимальный baseline:

- версия проекта;
- версия Docker-образа Collabora;
- список поддерживаемых языков;
- для каждого языка - цепочка кодов: код языка в приложении -> код, который приложение передает редактору в параметре `lang` -> файл каталога, который реально загрузится;
- список приложений: Writer, Calc, Impress, при необходимости Draw;
- список режимов интерфейса, если есть несколько дизайнов;
- активные `bundle.js`, `l10n/ui-*.json`, `l10n/uno/*.json`, `l10n/locore/*.json`;
- LibreOffice resources: `.po`, `.mo`, `.ui`, если доступны;
- набор тестовых документов для Writer, Calc, Impress;
- список UI-состояний: вкладки ribbon, системные меню внутри приложения, context menu, sidebar, dialogs, tooltips, формулы, таблицы, диаграммы, сводные таблицы, картинки, фигуры.

Отдельно нужно зафиксировать, как код языка превращается в загруженный каталог. Цепочка такая: пользователь выбирает язык в приложении (например `ru`) -> приложение передает редактору код в параметре `lang` (в этом проекте маппинг лежит в `co_lang`, `ru` -> `ru-RU`) -> редактор нормализует регион и грузит конкретный файл (`ru-RU` -> `ui-ru.json`). Если код неправильный или с лишним регионом, редактор может молча не загрузить каталог или загрузить чужой - тогда весь язык покажется непереведенным, хотя каталог на диске есть. Неверный код иногда меняет и отдельные строки, например подпись языка в статусной строке или сырой placeholder вроде `{ru}`. Это частая и незаметная причина нулевого покрытия, поэтому цепочку надо проверить для каждого языка отдельно.

Команда агенту:

```text
Ты локализационный аудитор. Работай только read-only.

Задача: зафиксируй baseline локализации проекта Collabora Online.

Контекст:
- Проект уже запущен локально.
- Нужно покрыть все поддерживаемые языки проекта.
- Нужно покрыть Writer, Calc, Impress и общие элементы интерфейса.
- Ничего не меняй.

Результат:
1. Список найденных языков и где они заданы.
2. Список найденных источников локализации.
3. Список приложений и UI-состояний, которые надо будет проверять.
4. Список файлов/образов/контейнеров, из которых взят baseline.
5. Риски и неизвестные места.

Контроль полноты:
- Все языки должны совпасть со списком проекта.
- Должно быть явно указано, что не покрыто.
- Не должно быть правок в файлах проекта, кроме отчета, если я отдельно разрешу его создать.
```

Контроль, не основанный на доверии агенту:

### Этап 2. Статическая инвентаризация строк

На этом этапе агент читает код и каталоги, но еще не доказывает рендер.

Нужно собрать:

- все прямые `_()` строки;
- все `_UNO(...)` команды;
- все строки frontend i18n;
- все строки из `ui-*.json`;
- все строки из `uno/*.json`;
- все строки из `locore/*.json`;
- все доступные PO/MO/UI ресурсы LibreOffice;
- все места, где строка встречается в коде.

Ключевой результат - occurrence map.

Минимальные поля occurrence map:

| Поле | Зачем нужно |
| --- | --- |
| `msgid` | Английская строка |
| `source_class` | `ui-json`, `uno-json`, `lo-core-mo`, frontend i18n, unresolved. Может быть несколько значений сразу: какой слой реально рисует строку, указывает `render_path` |
| `app` | Writer, Calc, Impress, shared |
| `file` | Файл или bundled segment |
| `line_or_offset` | Где найдено |
| `call_type` | `_()`, `_UNO(...)`, frontend `t()`, generated |
| `control_type` | tab, group, button, tooltip, menu, dialog, message |
| `neighbor_text` | Соседние слова или фраза |
| `command_id` | UNO command id или control id, если есть |
| `is_fragment` | Похоже ли на кусок фразы (детектится анализом call-site: `_("x")+var`, шаблоны) |
| `render_path` | Чем строка реально отрисована: `client-_()`, `server-jsdialog-core`, `uno`. Подтверждается экспериментом (правка источника → rebuild/cache-bust → изменился ли рендер), не предположением |
| `notes` | Что надо проверить вручную |

Поле `render_path` нельзя заполнять догадкой по тому, в каком каталоге лежит строка. Его подтверждают экспериментом: поменять строку в предполагаемом источнике, пересобрать образ, сбросить кэш браузера и посмотреть, изменился ли экран. Изменился - источник верный; нет (как с `Clipboard`, который лежит в `ui-ru.json`, но рисуется ядром) - источник другой, и правка этого каталога бесполезна.

Когда сопоставляешь строку с экрана со строкой из каталога, ее нельзя сравнивать буквально. Перед сравнением обе строки надо привести к одному виду:

- убрать маркеры мнемоник (`~`, `_`): `~Save` и `Save` - одна и та же строка;
- убрать хвостовое многоточие (`…` или `...`);
- убрать подсказку горячей клавиши в конце, вроде `(Ctrl+S)`;
- убрать хвостовое двоеточие;
- строки с подстановками (`{1}`, `%1`) сравнивать как шаблон через регулярное выражение, а не как точный текст.

Без этой нормализации одна и та же строка выглядит как разные, и появляется куча ложных "не переведено". Для строк ядра в `.mo` каталог сначала надо распаковать в текст (`msgunfmt`, нужен пакет gettext), иначе сравнивать не с чем.

Таблицу coverage из этого этапа нельзя считать фактом. Она строится по наличию строки в каталоге: если перевод в каталоге есть, строка считается переведенной. Но наличие в каталоге не значит, что именно он рисует текст (тот же `Clipboard`: лежит в `ui-ru.json`, а на экране английский, потому что рисует ядро). Поэтому процент из coverage - это верхняя оценка, гипотеза, а не правда; правду дает только рендер-проход (Этап 3). Обязательный шаг: взять выборку строк, которые по coverage числятся переведенными, и сверить с реальным экраном - так станет видно, насколько каталог расходится с рендером.

### Этап 3. Рендер-инвентаризация через браузер

Статика показывает, что может быть использовано. Но она не доказывает, что это реально видно пользователю.

Рендер-инвентаризация должна пройти по всем целевым языкам и состояниям UI.

Что проверять:

- стартовый экран редактора;
- все вкладки ribbon;
- dropdown-меню;
- контекстные меню;
- sidebar;
- dialogs;
- tooltips;
- статусную строку;
- sheet tabs в Calc;
- функции и формулы в Calc;
- таблицы и сводные таблицы;
- изображения, фигуры, диаграммы;
- презентационные режимы Impress;
- системные меню внутри Collabora, если они являются частью web UI.

Важно: браузерные или OS-native меню, которые приложение не контролирует, надо пометить как out of scope, а не пытаться переводить через Collabora.

Снять текст со страницы сложнее, чем просто прочитать `textContent`:

- всплывающие подсказки (tooltips) видны только при наведении - их ловят отдельным hover-проходом, в статическом DOM их нет;
- диалоги рисуются не обычным HTML, а деревом виджетов (JSDialog) - текст достают оттуда;
- часть диалогов живет во вложенных iframe того же origin - надо заходить внутрь;
- сетка таблицы и слайд - это `<canvas>`, текста там нет вообще; такой текст вне DOM и вне scope.

Чтобы поверхность реально открылась, мало кликать программно. Меню и диалоги редактора рисует сервер, и они реагируют только на настоящие события ввода (реальные мышь и клавиатура Playwright), а не на события, разосланные из JavaScript - синтетический правый клик или hover их не откроют. Кроме того, многие диалоги открываются только из подготовленного состояния: вставленная картинка, диаграмма или таблица, поставленный в текст курсор, выделенная ячейка. Если состояние не подготовить, кнопка не откроет ничего.

Самое легко пропускаемое (проверять отдельным списком):

- контекстные меню на canvas: заголовок столбца, заголовок строки, ярлычок листа, фигура;
- подменю, которые надо раскрывать отдельным шагом;
- диалоги, доступные только при активном выделении.

Еще две вещи. Кэш редактора - это ловушка: ассеты кэшируются надолго под именем, которое не меняется от наших правок, поэтому "на экране старое" может быть не отсутствием перевода, а просто кэшем. При этом обычный сброс кэша спасает не полностью: часть файлов сервер подставляет в обход нашего параметра обновления (например скрипт брендирования), а каталоги `ui-<loc>.json` грузятся в рантайме и тоже минуют сброс на уровне HTML. Поэтому надежнее всего проверять рендер в чистом профиле браузера (без старого кэша) на каждый прогон и дополнительно сверять отданный файл запросом мимо кэша (`curl` или `fetch(url, {cache: 'reload'})`) - два независимых сигнала. И отдельно надо честно фиксировать то, что не удалось открыть (`notReached`): если до какого-то меню или диалога не дошли, это записывается явно с причиной по каждому пункту, а измеренный процент покрытия считается верхней оценкой, а не фактом.

Команда агенту:


### Этап 4. Карта коллизий

После статической и рендер-инвентаризации нужно понять, какие строки безопасны.

Классы риска:

| Класс | Что значит | Можно ли автоматически переводить |
| --- | --- | --- |
| `single-use` | Один `msgid`, один контекст | Обычно да, после проверки |
| `same-context multi-use` | Несколько мест, но смысл один | Обычно да, если перевод одинаково подходит |
| `mixed-context` | Несколько мест с разным смыслом | Нет, нужен разнос или ручное решение |
| `composed-fragment` | Кусок составной фразы | Нет, надо переписывать на цельную фразу |
| `source-mismatch` | Видно в UI, но источник не тот, что ожидался | Нет, сначала выяснить источник |
| `core-only` | Строка приходит из LibreOffice-core | Через PO/MO/UI, не через `ui-json` |
| `technical-name` | Формат, функция, бренд, имя команды | Часто надо оставить английским или проверить терминологию |
| `layout-risk` | Перевод может не поместиться или сломать RTL/bidi | Только с визуальной проверкой |
| `render-path-mismatch` | Строка есть в ожидаемом каталоге, но рисуется из другого слоя (как `Clipboard`) | Нет, правка каталога не помогает; чинить тот слой, что реально рендерит |
| `plural-form` | Число-зависимые формы (1/2/5 в ru, 6 форм в ar) | Через gettext plural в ядре; плоский клиентский JSON их толком не выражает |

Команда агенту:

```text

Контекст:
- Нельзя предлагать перевод без учета всех occurrence.
- Нужно учитывать языки с разной грамматикой и RTL.
- Нужно отдельно выделить составные фразы.

Результат:
1. Таблица collision map.
2. Для каждого msgid: risk_class, количество occurrence, список контекстов, affected locales.
3. Список safe candidates для автоматического доперевода.
4. Список hard cases для человека.
5. Для hard cases указать, почему простая JSON-правка опасна.

Контроль полноты:
- Каждый msgid из occurrence map должен попасть ровно в один risk_class.
- Для каждого mixed-context должна быть ссылка на разные контексты.
- Для каждого composed-fragment должна быть показана исходная сборка фразы.
```

Контроль, не основанный на доверии агенту:

- Collision map должна покрывать все `msgid` из occurrence map. Не только проблемные.
- Каждая строка collision map должна ссылаться на список `occurrence_id`. Если id нет, класс риска не доказан.
- Для `single-use` должно быть ровно одно occurrence id.
- Для `same-context multi-use` должно быть несколько occurrence id и объяснение, почему контекст одинаковый.
- Для `mixed-context` должны быть перечислены разные context groups.
- Для `composed-fragment` должен быть указан fragment pattern: какие куски соединяются и где это видно.
- Для `technical-name` надо указать, почему английский может быть допустим: формат файла, API-name, функция, бренд, общепринятый термин.

Независимые проверки:

- Сравнить set(`msgid`) из `occurrences.json` и set(`msgid`) из `collisions.csv`. Разница должна быть пустой.
- Сравнить сумму `occurrence_id` по collision map с occurrence map. Каждый occurrence должен быть классифицирован.
- Выбрать все строки с `occurrence_count > 1` и проверить, что они не попали автоматически в `single-use`.
- Выбрать все строки, где `neighbor_text` различается, и проверить, что агент объяснил, почему это same-context или mixed-context.
- Проверить, что rendered leftovers тоже получили risk class, а не остались только в QA-отчете.

Признаки частичного выполнения:

- Collision map содержит только подозрительные строки, а остальные исчезли.
- Risk class есть, но occurrence ids отсутствуют.
- Все строки с несколькими occurrence автоматически помечены safe.
- Агент не выделил composed fragments.
- Агент не показывает разные context groups для mixed-context.

Как добивать:

```text
Collision map неполная. Доработай только классификацию.

1. Возьми полный set msgid из occurrences.json.
2. Возьми текущий collisions.csv.
3. Построй missing_collision_rows.csv.
4. Для каждой отсутствующей строки добавь risk_class.
5. Для каждой строки с occurrence_count > 1 добавь context_groups.
6. Не предлагай переводы на этом шаге.

Критерий завершения: set(msgid) в occurrences.json полностью равен set(msgid) в collisions.csv.
```

### Этап 5. Подготовка переводов

Перевод надо готовить не как свободный текст, а как контролируемую таблицу.

Для каждого кандидата:

- английский `msgid`;
- контекст;
- source class;
- текущий перевод по каждому языку;
- предлагаемый перевод;
- источник предложения: существующий LO-перевод, текущий каталог, терминологическая база, ручной перевод, ИИ;
- статус: safe, needs human, rejected, accepted;
- примечания по грамматике, длине, RTL.

Нельзя просить агента "переведи все". Надо просить "предложи кандидаты, но не вноси их".

Команда агенту:

```text
Ты переводческий ассистент для UI-локализации.

Задача: подготовить кандидаты переводов, но не менять файлы.

Контекст:
- Используй только строки risk_class single-use и same-context multi-use.
- Не трогай mixed-context, composed-fragment, core-only и unresolved.
- Для каждого языка дай перевод с учетом UI-контекста.
- Если не уверен, ставь needs human, а не угадывай.

Результат:
1. Таблица translation candidates.
2. Для каждого msgid: контекст, source class, affected locales, proposed translation, confidence, reason.
3. Список строк, которые нельзя переводить автоматически.
4. Список терминов, где нужен единый глоссарий.

Контроль полноты:
- В таблицу не должны попасть hard cases.
- Для каждого перевода должна быть причина.
- Для каждого языка должен быть статус: proposed, existing, needs human, keep English.
```

Контроль, не основанный на доверии агенту:

- `translation_candidates.csv` должен ссылаться на `msgid`, `risk_class`, `occurrence_id`, `source_class` и `locale`.
- Каждая строка должна иметь статус. Пустой статус означает, что работа не завершена.
- Перевод без `reason` и `source` не принимается. Source может быть `existing-locore`, `existing-uno`, `LibreOffice-po`, `glossary`, `human`, `ai-proposed`.
- Для `ai-proposed` нужен более низкий уровень доверия, пока человек не утвердил.
- Нельзя смешивать "предложено" и "внесено". Candidate table не должна сама менять файлы.
- Если перевод длинный, нужен layout note: возможный overflow, нужен screenshot, нужен shortened variant.
- Для RTL нужен note: порядок placeholders, punctuation, directionality.

Независимые проверки:

- Проверить, что все candidate rows имеют risk class `single-use` или `same-context multi-use`, если задача была только про safe candidates.
- Проверить, что нет строк с source class `lo-core-mo` в batch для `ui-json`.
- Проверить, что каждый target locale имеет статус по каждому candidate.
- Проверить, что glossary terms используются одинаково.
- Выбрать 20 случайных переводов и проверить по occurrence context, что перевод подходит именно туда.

Признаки частичного выполнения:

- Агент перевел только русский, хотя задача про все целевые языки.
- Агент заполнил только missing, но не отметил existing/keep English.
- Нет provenance перевода.
- Нет статуса `needs human`; агент сделал вид, что уверен во всем.
- В safe table попали mixed-context или unresolved строки.

Как добивать:

```text
Таблица переводов неполная. Не вноси патч.

1. Возьми approved safe candidates из collisions.csv.
2. Построй матрицу msgid x locale.
3. Для каждой ячейки укажи status: existing, proposed, needs human, keep English.
4. Для proposed укажи translation_source и reason.
5. Для needs human укажи конкретный вопрос человеку.
6. Не меняй файлы локализации.

Критерий завершения: в матрице нет пустых ячеек, и ни одна hard-case строка не попала в proposed без human approval.
```

### Этап 6. Разбор трудных случаев

Для трудных случаев агент должен готовить пакет решения для человека.

В пакете должны быть:

- где строка видна;
- все occurrence;
- почему есть коллизия;
- какие языки страдают;
- варианты решения;
- плюсы и минусы каждого варианта;
- что надо поменять в коде или каталогах;
- как проверить.

Вариантов может быть много, но лучше ограничивать их. Если простого решения нет, можно попросить до 20 вариантов, но с ранжированием.

Команда агенту:

```text
Ты архитектор локализации. Ничего не меняй.

Задача: подготовить decision package для hard cases.

Контекст:
- У нас есть msgid с коллизиями или составными фразами.
- Простое добавление перевода в JSON может сломать другие места.
- Нужно предложить варианты точечного решения.

Для каждого hard case дай:
1. Английский msgid.
2. Все места употребления.
3. Почему перевод конфликтует.
4. Какие языки затронуты.
5. До 20 вариантов решения, но сначала самые реалистичные.
6. Плюсы и минусы каждого варианта.
7. Какой вариант ты рекомендуешь и почему.
8. Какие тесты и скриншоты нужны после выбора.

Контроль полноты:
- Нельзя предлагать вариант без списка затронутых мест.
- Нельзя менять файлы без выбора человека.
- Если нужен новый контекстный key/helper, объясни fallback для английского.
```

Контроль, не основанный на доверии агенту:

- Для каждого hard case должен быть `decision_id`, чтобы потом patch ссылался не на свободный текст, а на утвержденное решение.
- Каждый вариант решения должен перечислять affected occurrence ids.
- Если вариант предлагает split key, он должен показать новую карту: old occurrence ids -> new key ids.
- Если вариант предлагает PO/MO/UNO правку, он должен показать, почему browser `ui-json` не является правильным источником.
- Если вариант оставляет английский, он должен объяснить, почему это приемлемо: термин, бренд, техническое имя, нет безопасного контекста.
- Каждый вариант должен иметь список новых рисков: новые коллизии, fallback, layout, RTL, migration, rollback.

Независимые проверки:

- Проверить, что все occurrence hard case покрыты хотя бы одним вариантом.
- Проверить, что recommended option не покрывает только "легкую" часть occurrence.
- Проверить, что split keys не создают технических fallback-строк в английском UI.
- Проверить, что для каждого варианта есть verification plan.
- Проверить, что человек явно выбрал вариант до любых правок.

Признаки частичного выполнения:

- Агент предлагает один вариант без альтернатив.
- Агент не показывает affected occurrence ids.
- Агент говорит "лучше разделить ключ", но не показывает, какие ключи и какие места.
- Агент предлагает править `ui-json` для core/UNO строки.
- Агент не описывает rollback.

Как добивать:

```text
Decision package неполный. Не меняй файлы.

Для каждого hard case добавь:
1. decision_id.
2. Полный список occurrence_id.
3. Минимум 3 варианта решения, если есть больше одного разумного пути.
4. Для каждого варианта: affected occurrence ids, новые keys/resources, плюсы, минусы, новые риски, rollback.
5. Рекомендуемый вариант отдельно, но без реализации.

Критерий завершения: человек может выбрать вариант, не задавая уточняющих вопросов о том, где строка встречается и что сломается.
```

## Как делать точечные правки

### Безопасный direct `ui-json`

Можно править, если:

- строка относится к `ui-json`;
- у нее `single-use` или `same-context multi-use`;
- перевод одинаково подходит всем occurrence;
- нет признаков составной фразы;
- перевод не ломает ширину/RTL;
- есть план рендер-проверки.

В этом проекте такие правки кладут как данные в `editor/l10n/overrides/client/<lang>.json` — их мёрджит deployed-сборка `editor/Dockerfile.online` (`editor/Dockerfile` — лишь stopgap/fallback), а не вручную меняют уже собранный файл в контейнере. Тогда правка воспроизводима при пересборке.

Команда агенту:

```text
Ты исполнитель точечной localization-правки.

Задача: внести только safe direct ui-json переводы.

Контекст:
- Разрешены только msgid из утвержденного списка safe candidates.
- Нельзя трогать mixed-context, composed-fragment, uno-json, lo-core-mo.
- Класть переводы как данные в editor/l10n/overrides/client/<lang>.json (их мёрджит deployed-сборка editor/Dockerfile.online; editor/Dockerfile — stopgap).

Результат:
1. Минимальная правка.
2. Список добавленных ключей по языкам.
3. Обновленный WAL/update-log.
4. Команды проверки.

Контроль полноты:
- Каждый добавленный ключ должен быть из утвержденного списка.
- Не должно быть массовой замены каталогов.
- После сборки должно быть доказано, что ключ реально появился в активном `ui-*.json`.
```

Контроль, не основанный на доверии агенту:

- До правки должен существовать approved list: `approved_ui_json_candidates.csv`.
- Diff должен менять только разрешенные файлы и только разрешенные ключи.
- После сборки надо извлечь активный `ui-<locale>.json` из контейнера и проверить, что ключ реально присутствует там, а не только в исходном патче.
- Нужно проверить, что значение не перезаписало существующий более свежий перевод.
- Нужно проверить, что ключ не добавлен в язык, где он уже был переведен иначе.
- Для каждого добавленного ключа нужен rendered scenario или причина, почему рендер невозможен на этом batch.

Независимые проверки:

- Сравнить set(keys changed in diff) с set(approved candidates). Разница должна быть пустой.
- Сравнить active container catalog before/after.
- Проверить, что `same-as-English` не стало ошибочно считаться translated, если перевод совпадает с английским намеренно.
- Проверить, что UI не показывает технические ключи.

Если агент сделал слишком широкий патч:

```text
Патч шире утвержденного scope. Откати только свои изменения этого batch и пересобери patch заново.

Разрешено менять только keys из approved_ui_json_candidates.csv.
Выведи таблицу:
- approved key
- changed file
- old value
- new value
- rendered scenario id

Критерий завершения: git diff содержит только разрешенные ключи и update-log описывает только этот batch.
```

### Контекстный разнос в коде

Если один `msgid` конфликтует в разных местах, нельзя просто добавить один перевод.

Возможные решения:

1. Программный контекст (`msgctxt`/`pgettext`) есть только в ядре (`.po`/`.mo`). В клиентских каталогах (`ui/uno/locore-*.json`) его нет - это плоские словари "английская строка -> перевод", по одному значению на строку. Смысловой (де-факто) контекст у строки есть, а технического разделителя контекста на клиенте нет. Поэтому развести омоним на клиенте можно только разными ключами (правка вызова в коде), а настоящий `msgctxt` доступен лишь на уровне ядра.
2. Если контекст не поддерживается, добавить контекстный helper, который ищет скрытый контекстный ключ, но для английского fallback показывает нормальную английскую строку.
3. Если строка составная, заменить куски на цельную фразу с placeholders.
4. Если строка приходит из UNO/core, не разносить ее в browser JSON, а чинить источник UNO/LibreOffice.

Плохой вариант:

```text
_("Line|shape-tool")
```

если при отсутствии перевода пользователь увидит `Line|shape-tool`.

Лучше иметь fallback:

```text
translateContext("shape-tool", "Line")
```

где английский fallback остается `Line`, а перевод может использовать контекстный ключ.

Команда агенту:

```text
Ты разработчик контекстной локализации.

Задача: предложить минимальный способ разделить конфликтующий msgid без поломки английского fallback.

Контекст:
- Один английский текст используется в нескольких разных смыслах.
- Простая запись в ui-json небезопасна.
- Нужно сохранить нормальный английский интерфейс, если перевода нет.

Результат:
1. Предложение API/helper или использование существующего механизма контекста.
2. Список мест, где надо заменить вызовы.
3. Новые контекстные ключи.
4. Fallback behavior.
5. План миграции переводов.
6. Тесты и рендер-проверки.

Контроль полноты:
- Все старые occurrence должны быть распределены по новым контекстам.
- Английский fallback не должен показывать технические ключи.
- Не должно появиться новых коллизий между новыми ключами.
```

Контроль, не основанный на доверии агенту:

- Перед кодовой правкой должна быть таблица `split_plan.csv`: old msgid, old occurrence ids, new key, new context, fallback.
- После правки надо пересобрать occurrence map и доказать, что старые occurrence исчезли или распределились по новым ключам.
- Нужно проверить английский UI: если перевода нет, пользователь должен видеть нормальный английский текст, а не `Line.shapeTool` или `shape-tool|Line`.
- Нужно проверить, что новые ключи не совпали с уже существующими ключами другого смысла.
- Нужно проверить, что placeholder order работает для языков с другим порядком слов.

Независимые проверки:

- Сравнить old occurrence ids из decision package с new occurrence ids после правки.
- Проверить, что каждый new key имеет ровно тот context, ради которого создан.
- Проверить, что old msgid больше не используется в конфликтном месте.
- Прогнать rendered scenarios для всех старых мест, а не только для одного.

Если агент сделал неполный split:

```text
Контекстный split неполный. Не добавляй новые переводы.

1. Возьми split_plan.csv.
2. Для каждого old occurrence id покажи, куда он попал после правки.
3. Построй missing_or_unsplit_occurrences.csv.
4. Исправь только эти места.
5. Пересобери occurrence map.

Критерий завершения: все old occurrence ids из split_plan имеют new key или явно documented keep-old reason.
```

### UNO и LibreOffice-core

Если строка относится к `_UNO(...)` или LibreOffice-core, надо быть особенно осторожным.

Правильный путь:

- найти исходный UNO command или LO resource;
- проверить, есть ли PO-контекст;
- править PO, а не бинарный `.mo`, если есть такая возможность;
- собрать `.mo` через gettext toolchain;
- пересобрать образ или аккуратно заменить ресурс в image build;
- проверить, что Collabora реально использует новый ресурс;
- проверить все команды, которые могли использовать тот же английский текст.

Учти: `gettext` (`msgfmt`/`msgunfmt`) обычно не входит в образ - его надо доустановить. Делать это в Dockerfile сборки, а не вручную в работающем контейнере: в работающем контейнере все пропадет при пересоздании.

Команда агенту:

```text
Ты специалист по LibreOffice/UNO локализации.

Задача: подготовить план исправления строки из uno-json или lo-core-mo.

Контекст:
- Простая правка ui-json запрещена.
- Нужно найти правильный источник: UNO map, PO, MO, UI resource или generator.
- Нужно не ломать другие команды с таким же английским текстом.

Результат:
1. Где строка определена.
2. Какие команды или диалоги ее используют.
3. Есть ли контекст в PO/LO source.
4. Какой файл надо менять.
5. Как пересобрать ресурс.
6. Как проверить в активном образе.
7. Какие соседние строки могут пострадать.

Контроль полноты:
- Не предлагать редактировать .mo вручную как основной путь.
- Не предлагать правку ui-json, если UI берет строку из core.
- Указать rollback-план.
```

Контроль, не основанный на доверии агенту:

- План должен показывать source chain: visible text -> UNO command/dialog -> source catalog -> built artifact -> active container file.
- Если агент не может показать source chain, правку делать нельзя.
- Для `.po`/`.mo` нужно проверить msgctxt или эквивалентный контекст. Если один `msgid` встречается в нескольких контекстах, нельзя менять все в одном месте без анализа.
- После сборки нужно проверить не только файл `.mo`, но и видимый UI, потому что Collabora может брать строку из generated UNO JSON или другого кеша.
- Нужно проверить все команды, где тот же английский текст встречается в UNO/core, а не только один экран.

Независимые проверки:

- `msgunfmt` или другой gettext-инструмент должен показать ожидаемый `msgid`/`msgstr` в итоговом `.mo`.
- Hash итогового `.mo` должен отличаться от baseline только там, где ожидается.
- Active container должен содержать новый `.mo` или regenerated `uno/*.json`.
- Rendered inventory должен подтвердить изменение в нужном месте.
- Collision map должна быть обновлена, если PO-context разделяет смысл.

Если агент предложил сомнительный core-патч:

```text
План для UNO/core неполный. Не меняй файлы.

Дай source chain для строки:
1. visible text и screen state;
2. command/dialog id;
3. source PO/UI file или generator;
4. build command для .mo/uno json;
5. active container path;
6. все другие occurrences этого msgid в UNO/core;
7. rollback.

Критерий завершения: можно объяснить, почему правка не затронет другой смысл той же английской строки.
```

## Как контролировать агента

Агент полезен, но его надо ограничивать.

Хорошие правила:

- Сначала read-only анализ, потом правки.
- Любая правка должна ссылаться на конкретный artifact: occurrence map, collision map, approved candidates.
- Не разрешать массовый импорт свежих PO без сравнения версий и контекстов.
- Не разрешать "переведи все" без risk classification.
- Требовать отчет, где числа сходятся.
- Требовать список того, что не покрыто.
- Требовать rendered proof для каждого измененного UI-сценария.
- Требовать отдельный список false positives.
- Требовать объяснение, почему конкретный source class чинится именно этим способом.
- Держать правки маленькими партиями.

Плохая команда агенту:

```text
Переведи все английские строки на русский.
```

Хорошая команда агенту:

```text
Возьми только утвержденные safe candidates из translation_candidates.md.
Внеси переводы только для source_class=ui-json.
Не трогай строки с risk_class mixed-context, composed-fragment, core-only или unresolved.
После правки пересобери образ, проверь активные l10n файлы, открой Writer/Calc/Impress на ru/uk/ar/zh-CN и приложи rendered evidence.
Если любой шаг не получается, остановись и дай отчет, не угадывай.
```

### Жесткий протокол работы с агентом

Чтобы агент не сделал 30% задачи и не выдал это за полный результат, задачу надо давать как контракт:

1. Scope фиксируется до работы.
2. Агент обязан создать manifest.
3. Агент обязан заполнить raw-таблицу.
4. Агент обязан вывести проверочные counts.
5. Агент обязан перечислить skipped/failed.
6. Агент обязан явно написать, что не покрыто.
7. Человек или независимая проверка сравнивает counts и artifacts.
8. Только после этого результат считается принятым.

Формула хорошей задачи:

```text
Роль:
Ты не переводчик, а аудитор локализации. Твоя цель - доказуемо собрать полный artifact.

Scope:
Перечисли включенные языки, приложения, файлы, UI-состояния. Не расширяй и не сужай scope без отдельного списка changes.

Forbidden:
Не делай summary-only output. Не делай sample. Не пропускай failed/skipped. Не вноси правки.

Required artifacts:
1. machine-readable manifest;
2. full CSV/JSON table;
3. Markdown summary;
4. counts/invariants;
5. list of gaps/skips/unresolved;
6. reproduction commands.

Acceptance:
Работа принимается только если counts сходятся и все expected rows присутствуют.
Если не успеваешь или не можешь покрыть все, остановись и выдай missing_work.csv.
```

### Как ловить частичное выполнение

Самые частые признаки, что агент не сделал полный проход:

- "Основные языки" вместо полного списка языков.
- "Ключевые экраны" вместо manifest всех экранов.
- "Например" в таблице, где нужен полный список.
- "Top N" без полной выгрузки.
- Нет строки для failed/skipped.
- Нет raw JSON/CSV.
- Нет hash входных файлов.
- Нет способа повторить extraction.
- Нет связи между summary и raw rows.
- Нет ссылок от patch к approved decision/candidate.
- Нет before/after evidence.

Если такое найдено, не надо просить "проверь внимательнее". Это почти всегда даст еще одно красивое summary. Надо давать delta-задачу:

```text
Ты сделал неполный результат. Не повторяй summary.

Сделай только недостающий artifact:
- файл: missing_work.csv
- вход: текущий artifact X
- проверка: invariant Y
- критерий завершения: set(A) == set(B)

Никаких новых выводов, пока artifact не построен.
```

### Как заставлять дорабатывать до полного результата

Рабочая стратегия:

1. Сначала попросить агента построить expected set.
   Например: все языки, все приложения, все сценарии, все `msgid`.

2. Потом попросить построить actual set.
   Например: что реально проверено, что реально есть в таблице, какие screenshots сохранены.

3. Потом попросить построить gap set.
   Например: `expected - actual`.

4. Потом запускать только gap set.
   Это не дает агенту снова делать выборочный общий проход.

5. После доработки снова пересчитать invariant.

Пример команды:

```text
Не делай повторный общий аудит.

1. Прочитай expected_scenarios.csv.
2. Прочитай run_manifest.json.
3. Построй missing_scenarios.csv как expected - actual.
4. Прогони только missing_scenarios.csv.
5. Обнови run_manifest.json.
6. Выведи counts: expected, before_actual, missing_before, added, missing_after.

Критерий завершения: missing_after = 0 или каждая оставшаяся строка имеет статус blocked с причиной.
```

### Когда можно доверять выводу агента

Доверять можно не "агенту", а артефактам.

Вывод можно принимать, если:

- есть полный expected set;
- есть полный actual set;
- есть gap set;
- есть machine-readable rows;
- есть воспроизводимая команда;
- counts сходятся;
- skipped/failed не скрыты;
- raw rows можно проверить выборочно;
- patch ссылается на approved ids;
- rendered proof покрывает измененные states.

Если хотя бы один из этих пунктов отсутствует, результат надо считать предварительным.

## Как проверять полноту

Одного скриншота недостаточно.

Нужно несколько независимых сверок:

1. Static extraction: сколько строк найдено в коде.
2. Catalog coverage: сколько строк есть в каталогах по каждому языку.
3. Occurrence map: где каждая строка используется.
4. Rendered inventory: что реально видно в браузере.
5. Collision map: где один английский текст имеет разные смыслы.
6. Patch coverage: какие строки реально изменены.
7. Rendered regression: что изменилось после патча.
8. Visual proof: скриншоты до/после.
9. False positives: английские строки, которые должны остаться.

Контрольные вопросы:

- Все ли 22 языка прошли одинаковый набор сценариев?
- Есть ли строки, которые видны в браузере, но отсутствуют в статике?
- Есть ли строки, которые есть в статике, но не встречаются в рендере?
- Все ли измененные ключи имеют occurrence map?
- Нет ли одного перевода, примененного к разным смыслам?
- Нет ли составных фраз из отдельных слов?
- Не сломались ли RTL языки?
- Не вылезли ли подписи из кнопок?
- Не появились ли технические ключи вместо английского fallback?
- Не пропали ли icons/tooltips/menus после правки?

Полнота и качество - разные вещи. Все проверки выше отвечают на вопрос "есть ли перевод и виден ли он", но не на вопрос "правильный ли он". Сходящиеся числа (инварианты) доказывают, что ничего не пропущено и результат воспроизводим, но не доказывают, что перевод корректен. Качество проверяется отдельным проходом сильной модели, хорошо знающей целевой язык, прямо по интерфейсу (по рендеру/скриншотам, с учетом контекста). А финальную полировку лучше делать носителем языка - точнее, носителем-предметником, который знает доменную терминологию.

### Проверки по типам работы

| Работа | Что агент обычно недоделывает | Как проверить независимо | Как добивать |
| --- | --- | --- | --- |
| Baseline | Берет часть языков или не тот контейнер | Сравнить locale set с кодом, пересчитать hash извлеченных файлов | Требовать `baseline_manifest.json` и `missing_catalogs.csv` |
| Static extraction | Дает summary вместо полного occurrence map | Пересчитать `_()`, `_UNO(...)`, i18n keys другим скриптом | Требовать `occurrences.json`, `extraction_stats.json`, `unresolved.csv` |
| Catalog coverage | Проверяет только `ui-*.json` | Сравнить `ui`, `uno`, `locore`, `.mo` отдельно | Требовать coverage matrix по каждому source class |
| Rendered inventory | Проверяет один язык или одну вкладку | Сравнить `scenario_manifest` и `run_manifest` | Требовать `missing_scenarios.csv` и delta-run |
| Collision map | Классифицирует только очевидные проблемы | Сравнить set(msgid) occurrence map и collision map | Требовать полное покрытие всех `msgid` |
| Translation candidates | Переводит все подряд | Проверить risk_class и source_class для каждой candidate row | Удалить hard cases из batch и требовать статусы по каждой locale |
| Direct ui-json patch | Меняет больше ключей, чем утверждено | Сравнить diff keys с approved candidates | Сузить patch до approved set |
| Context split | Разделяет часть occurrence, оставляет конфликтные места | Сравнить old occurrence ids с new occurrence ids | Требовать `missing_or_unsplit_occurrences.csv` |
| UNO/core patch | Предлагает не тот слой исправления | Проверить source chain до `.po/.mo/uno json` и rendered UI | Требовать source-chain report до правки |
| Verification | Показывает только удачные скриншоты | Проверить failed/skipped и before/after text dump | Требовать полный verification manifest |
| Upgrade maintenance | Переносит старые патчи вслепую | Сравнить old/new occurrence map и changed contexts | Требовать release diff matrix |

### Пример независимых инвариантов

Допустим, статический отчет утверждает:

```text
locales = 22
direct_ui_msgid = 436
referenced_uno_labels = 300
global_uno_labels = 813
```

Тогда должны существовать независимые проверки:

```text
count(locales_from_project) == 22
count(unique direct _() msgid in selected code segments) == 436
count(unique resolved UNO labels from referenced commands) == 300
count(unique labels in global UNO map) == 813
```

Если агент не может показать, как эти числа пересчитаны, это не audit, а пересказ.

### Как принимать частично заблокированную работу

Иногда полный проход реально невозможен: Playwright занят, fixture не открывается, источник minified, `.po` отсутствует, команда создается динамически.

Это нормально, но blocked-состояние должно быть явным.

Для каждого blocked item нужны поля:

| Поле | Что писать |
| --- | --- |
| `blocked_id` | Стабильный id |
| `scope_item` | Что именно не покрыто |
| `reason` | Почему не покрыто |
| `attempted` | Что уже пробовали |
| `needed` | Что нужно, чтобы закрыть |
| `risk` | Что может быть неверным из-за блока |
| `owner` | Агент, человек, внешний инструмент |
| `next_check` | Как вернуться к проверке |

Фраза "не удалось проверить" без этих полей не считается отчетом.

Для устойчивой поддержки нужны не только правки, но и документы.

Рекомендуемые файлы:

Это и есть проверяемая опора для решения, что и каким слоем чинить.

## Стратегия работы с исходниками Collabora

Цель проекта — **собирать из исходников Collabora Online** (сервер coolwsd + браузерный bundle), а не патчить готовый образ. **Ядро LibreOffice/Collabora Office из исходников не собираем** — берём его закреплённым готовым бинарём (engine-assets), а строки ядра правим как данные (`.po`→`.mo`). Причина: при глубокой локализации постоянно всплывает то, что чинится только в исходниках Online — склейки строк в JavaScript, контекстный API перевода, клиентские каталоги. А всё, что мы меняем в UI (тексты, размеры, дизайн), живёт именно в слое Online, не в ядре. Здесь важнее надёжность и контроль, чем «быстро в прод».

Быстрый путь, от которого мы уходим, обычно выглядит так:

```text
FROM collabora/code:latest
```

а дальше Dockerfile правит уже собранные `bundle.js`, `bundle.css`, `cool.html`, `l10n/ui-*.json` и ресурсы образа через `sed`, `cp`. Это допустимо как **временный костыль** (брендовая мелочь, срочный hotfix), но как основа для локализации не годится.

Почему так нельзя жить долго:

- `latest` меняется без контроля. Сегодня и завтра это могут быть разные байты.
- Minified `bundle.js` плохо подходит для контекстного разделения строк.
- `sed`-патч по собранному файлу легко ломается при любом upstream-изменении.
- Невозможно нормально добавить новый контекстный API локализации.
- Трудно понять, какой исходный файл породил строку, и перенести правку после обновления.
- Трудно доказать, что патч правит правильный слой, а не маскирует проблему.

Поэтому движемся по ступеням, но конечная точка одна — **сборка Online из закреплённых исходников** (ядро остаётся закреплённым готовым бинарём).

Версию фиксируем жёстко. Переводы делаются под **конкретную** версию: набор строк, их `msgid` и `msgctxt` версионно-зависимы. Закрепляем **пару**, и она должна совпадать: commit/тег Collabora Online И версию готового ядра/engine-assets (обе в `upstream.json`). Обновление версии — это отдельное контролируемое событие с приёмкой всех переводов, а не «подтянул latest».

Цену стоит назвать честно: сборка **Online** из исходников подъёмна — официально это `./configure && make` поверх готовых engine-assets, от минут до получаса, и проходит локально. А вот сборка **ядра** из исходников тяжёлая (десятки гигабайт, часы); её мы не делаем — см. ниже «Когда всё же собирать ядро LO».

### Ступень 1 (временная): убрать недетерминированность готового образа

Пока сборка из исходников не поднята, нельзя оставаться на `latest` — надо хотя бы убрать недетерминированность готового образа.

Нужно:

1. Перестать использовать `collabora/code:latest`.
2. Закрепить образ по версии или digest.
3. Записать pinned image в baseline.
4. Сохранять hash активных `bundle.js`, `ui-*`, `uno/*`, `locore/*`, `.mo`.
5. Любой отчет строить только против этого pinned baseline.

Пример:

```text
collabora/code@sha256:<digest>
```

или конкретный tag, если digest по какой-то причине нельзя использовать.

Команда агенту:

```text
Ты release-инженер локализации. Ничего не меняй.

Задача: проверить, воспроизводимо ли зафиксирована версия Collabora.

Результат:
1. Найди, какой image/tag/digest используется для editor.
2. Укажи, используется ли latest.
3. Укажи digest активного образа.
4. Предложи минимальную правку для pinning.
5. Укажи, какие baseline artifacts надо пересобрать после pinning.

Контроль полноты:
- Должен быть указан текущий Dockerfile/compose path.
- Должен быть указан текущий image id/digest.
- Если используется latest, это должно быть явно помечено risk=high.
- Нельзя менять Dockerfile без отдельного разрешения.
```

### Ступень 2 (промежуточная): патчи как единицы, а не длинные `sed`-цепочки

Пока образ всё ещё готовый, изменения нельзя держать как длинные `sed`-цепочки. Это промежуточный уровень по пути к сборке из исходников.

Лучше иметь:

```text
editor/
  Dockerfile
  patches/
    collabora-online/
      0001-contextualize-line-label.patch
      0002-fix-composed-open-list-label.patch
    libreoffice-core/
      0001-ru-pivot-field-translation.patch
  scripts/
    extract-active-collabora-assets.sh
    build-l10n-occurrence-map.py
    verify-active-catalogs.py
  manifests/
    upstream.json
    patchset.json
```

Даже если патч все еще применяется к собранному образу, он должен быть описан как patch unit:

- что меняет;
- почему это правильный source class;
- к какому upstream digest относится;
- какие occurrence ids закрывает;
- какие rendered scenarios проверяют результат;
- как откатить.

Это делает текущий repo управляемым: приложение остается здесь, а все editor-интеграционные патчи имеют явные границы.

### Целевой уровень: сборка из закреплённых исходников

Это рабочая основа проекта. Из исходников мы собираем **Collabora Online** (сервер coolwsd + браузерный bundle) — держим не копию, а fork или submodule закреплённого commit. **Ядро LibreOffice/Collabora Office из исходников НЕ собираем**: берём закреплённым готовым бинарём (engine-assets), а его строки `.mo` правим как данные (наши `.po`). Полный контроль над интерфейсом это даёт малой кровью — всё, что мы меняем в UI, живёт в слое Online.

Что какой слой позволяет менять:

- **Тексты** (меню, лента, тултипы, диалоги) — все: данные (клиентские `ui/uno/locore-<lang>.json` + `.mo` ядра из наших `.po`) плюс немного кода в Online для расклейки фраз и разведения омонимов. Сборка ядра не нужна.
- **Размеры элементов** — CSS (часть — JS). Без сборки ядра.
- **Дизайн элементов** (цвета, формы, отступы, иконки) — CSS + замена SVG (глубже — JS). Без сборки ядра; так уже сделана glass-тема.

Единственное, что требует сборки ядра, — отрисовка самого документа на «холсте» (сетка Calc, слайд, текст в ячейках). Это содержимое документа, а не элементы интерфейса (см. «Когда всё же собирать ядро LO»).

```text
third_party/
  collabora-online/      # git submodule или pinned fork, закреплённый commit (СОБИРАЕМ из исходников)
editor/
  engine/                # закреплённые engine-assets ядра LO (готовый бинарь, НЕ исходники)
  l10n/
    overrides/
      core/<lang>.po      # наши переводы строк ядра, с msgctxt → компилируются в .mo
      client/<lang>.json  # наши оверлеи клиентских строк
  patches/                # ТОЛЬКО изменения кода Online (контекстный API, развод склеек)
  build/                  # детерминированная сборка Online из исходников + наложение данных
  manifests/
    upstream.json
    patchset.json
```

Главное разделение — переводы отдельно от кода:

- **Переводы — это ДАННЫЕ.** Наши переводы живут как файлы: `.po` с `msgctxt` для строк ядра и JSON-оверлеи для клиентских строк. На сборке из исходников они детерминированно компилируются и мёржатся в `.mo` и каталоги. Такие файлы легко ревьюить, легко отдавать носителям языка и легко масштабировать на 30–40 языков.
- **Патчи — это только КОД** (новый контекстный API локализации, переписывание склеенных фраз на цельные с placeholders). Данные перевода и патчи кода не смешиваются в одном изменении.

Почему fork/submodule лучше простой копии:

- сохраняется upstream history; можно делать rebase/merge;
- видно, какие файлы изменены относительно upstream;
- проще проверить, что правка применима к новой версии;
- меньше риск случайно редактировать собранный (generated) вывод вместо исходника.

Куда отдавать наружу — важно не перепутать канал:

- **переводы отдаются в Weblate** (площадка переводов LibreOffice/Collabora), а не через GitHub PR;
- **изменения кода** (контекстный API, фиксы склеек) — обычным PR в Collabora Online или LibreOffice.

Отдельно: сборка из исходников означает, что **security-фиксы апстрима бэкпортишь ты сам**. Это ещё одна причина, почему обновление версии — контролируемое событие с приёмкой, а не «подтянул свежее и в прод».

### Какой слой чинить (ядро или клиент)

Collabora Online не содержит все строки сам. Часть интерфейса идёт из LibreOffice-core: `.ui`, gettext `.po/.mo`, метаданные UNO-команд, диалоги, sidebar. Поэтому ядро LO у нас закреплено как готовый бинарь (engine-assets), а его строки правим как данные (`.mo` из наших `.po`) — без сборки ядра.

Но перед каждой правкой всё равно надо доказать, в каком слое живёт строка, иначе правка уйдёт не туда (это та же render-path атрибуция):

- строка реально приходит из LO-core, а не из клиентского `ui-json`/`uno-json`;
- есть конкретный PO/UI/исходный файл;
- есть путь сборки до `.mo` или generated-ресурса;
- есть rendered scenario, который доказывает результат на экране.

Для строк клиента правим наши JSON-оверлеи, для строк ядра — наши `.po` с `msgctxt`. Слой выбирается доказательством, а не на глаз.

### Когда всё же собирать ядро LO из исходников

Почти никогда. Сборка ядра из исходников нужна только если надо менять **отрисовку самого документа** (как ядро рисует сетку Calc, слайд, текст в ячейках) или **C++-поведение ядра**. Это содержимое документа, а не элементы интерфейса — для текстов, размеров и дизайна UI оно не нужно.

Это самый тяжёлый путь: «несколько часов даже на самом быстром компьютере», образ распухает до ~30 ГБ; обычно так делает только CI самой Collabora. Показательно, что даже официальная сборка «CODE for Web» НЕ пересобирает ядро — она подкладывает готовые engine-assets. Поэтому полная сборка ядра — необязательная эскалация, а не наш дефолт.

### Что не надо делать

Не надо:

- просто копировать весь Collabora source в app repo без upstream связи;
- продолжать жить на `latest`;
- редактировать minified `bundle.js` как основной способ долгосрочных правок;
- массово импортировать свежие PO/JSON без occurrence/collision map;
- смешивать app frontend переводы, Collabora browser UI и LibreOffice-core в одном patch batch;
- держать переводы как код-патчи вместо файлов данных (`.po`/JSON-оверлеи);
- обновлять версию редактора в прод без полной приёмки переводов («тяп-ляп в прод»);
- принимать патч, который нельзя связать с конкретным upstream commit или image digest.

### Контроль полноты для source-стратегии

Для этой стратегии нужны отдельные артефакты:

| Артефакт | Что доказывает |
| --- | --- |
| `upstream.json` | Какая версия Collabora и LibreOffice используется |
| `patchset.json` | Какие патчи применяются, к чему они относятся, какие occurrence закрывают |
| `source-build.md` | Как собрать editor image из source или patch series |
| `source-diff-report.md` | Чем fork отличается от upstream |
| `generated-vs-source-map.json` | Как bundled/generated строки связаны с исходными файлами |
| `release-diff-matrix.csv` | Что изменилось при обновлении upstream |

Инварианты:

- editor image не должен быть `latest`;
- Online собирается из закреплённых исходников; ядро LO — закреплённый готовый бинарь/engine-assets; обе версии зафиксированы в `upstream.json` и **совпадают**;
- переводы хранятся как данные (`.po`/JSON-оверлеи), а не как код-патчи;
- каждый patch (только код) имеет upstream base commit, owner и reason;
- каждая запись перевода и каждый code-патч ссылается на occurrence ids;
- каждый source-level split обновляет occurrence map;
- каждое обновление версии проходит полную приёмку до промоушена в прод и пересчитывает release diff matrix;
- generated output не редактируется, если доступен source-level путь.

Команда агенту:

```text
Ты архитектор source-level стратегии для Collabora localization.

Задача: предложить, как текущему проекту перейти от патчей по готовому образу к воспроизводимой source-level схеме.

Контекст:
- Сейчас editor может строиться от готового Collabora image.
- Нужны аккуратные локализационные правки, включая context split и PO/MO.
- Нельзя просто вендорить огромный source без плана поддержки.

Результат:
1. Текущий способ сборки editor.
2. Риски текущего способа.
3. Минимальный pinning plan.
4. Patch-series plan для текущего repo.
5. Fork/submodule plan для Collabora Online.
6. Отдельное решение, нужен ли LibreOffice-core source.
7. Список новых artifacts.
8. Migration plan по шагам.

Контроль полноты:
- Должен быть указан текущий image/tag/digest.
- Должно быть явно сказано, какие правки остаются image-level, а какие должны стать source-level.
- Должен быть rollback plan.
- Должна быть стратегия обновления upstream.
- Должен быть способ проверить, что патчи применяются к правильной версии.
```

## Как поддерживать после обновлений Collabora

При обновлении Collabora нельзя просто переносить старый патч.

Та же осторожность относится к чужим переводам: Collabora - это не стоковый LibreOffice. У Collabora свои строки и свой нотбукбар, поэтому "взять свежий языковой пакет LibreOffice и подложить" вслепую опасно: часть строк не совпадет по идентификатору или контексту, а часть наложится неправильно. Любой импорт чужих переводов сверяется по версии и контекстам.

Обновление версии — это контролируемое событие с полной приёмкой, а не перенос одного патча. Порядок такой:

1. Поднять новую закреплённую версию **сначала в непрод-окружении**, не трогая прод: новый commit Collabora Online + **совпадающую** версию готового ядра/engine-assets.
2. **Пересобрать Online из исходников** новой версии (ядро берётся готовым бинарём/engine-assets, не пересобирается).
3. Прогнать **полную приёмку, а не smoke**:
   - пересобрать occurrence map и покрытие;
   - прогнать render-ground-truth (скриншот по каждому объекту) по **всем** целевым языкам;
   - сделать drift-diff всех строк интерфейса против предыдущей версии.
4. Разобрать, что сломало обновление: переехавшие или переименованные `msgid`, изменившийся `msgctxt`, новые английские строки, удалённые строки, вернувшиеся склейки фраз.
5. Пере-применить (rebase) наши данные переводов и код-патчи на новую версию и починить поломки.
6. **Только после прохождения полной приёмки** — промоушен в прод. Надёжность важнее скорости.

Эта приёмка — гейт: те же инварианты покрытия и render-drift-diff гоняются в CI на каждом подъёме версии и должны пройти **до** промоушена. Скриншот по каждому объекту плюс второй проход-проверка (см. «Принцип доказательности») — единственное основание считать обновление принятым.

Команда агенту:

```text
Ты release-аудитор локализации после обновления Collabora.

Задача: сравнить старый и новый baseline и сказать, какие локализационные патчи безопасно переносить.

Контекст:
- Старые патчи могли быть привязаны к старым source strings.
- Новые PO/JSON могут иметь другие контексты.
- Нельзя массово переносить без проверки.

Результат:
1. Таблица unchanged keys.
2. Таблица removed keys.
3. Таблица new keys.
4. Таблица changed occurrences.
5. Список патчей и записей перевода, которые можно оставить.
6. Список патчей и записей перевода, которые надо пересмотреть.
7. План полной приёмки (не smoke): render-ground-truth по всем языкам + drift-diff против прошлой версии.

Контроль полноты:
- Каждый старый patch key должен быть классифицирован.
- Каждый новый английский leftover должен быть связан с source class.
- Если контекст изменился, патч/перевод должен быть помечен needs review.
- Промоушен в прод только после прохождения полной приёмки.
```

## Масштабирование на много языков

Когда языков не 4, а 30-40, важно не повторять всю работу для каждого.

Работа делится на два уровня:

- Поверхность снимается один раз. Список всех мест, где встречается каждая строка (occurrence map), не зависит от языка - его строят однократно.
- Покрытие по языку считается данными. Для конкретного языка достаточно сверить строки с его каталогами и `.mo` - это чистый расчет по файлам, без браузера, и он масштабируется на любое число языков.

Но есть то, что данными не проверить: как перевод реально смотрится на экране - перенос строк, выход за кнопку, RTL-раскладка, контекстные коллизии. Это видно только в рендере, и проверять надо по каждому языку. Браузер обычно один на инстанс, поэтому он и есть узкое место. Решение - распараллелить: поднять несколько виртуальных машин, по одному языку на машину, каждая со своим инстансом редактора и браузера; тогда скрин-проходы идут параллельно.

## Практический порядок внедрения

Реалистичный порядок такой:

1. Закончить occurrence map.
2. Сделать rendered inventory через Playwright.
3. Построить collision map.
4. Выделить safe candidates.
5. Согласовать глоссарий для частых терминов.
6. Внести маленький batch safe `ui-json` переводов.
7. Пересобрать образ.
8. Проверить активные каталоги.
9. Прогнать Playwright smoke по всем языкам.
10. Прогнать полный Playwright по ключевым языкам: ru, uk, ar, zh-CN, плюс несколько европейских и RTL.
11. Исправить проблемы ширины, RTL, составных фраз.
12. Перейти к hard cases.
13. Для hard cases выбирать решение человеком.
14. Вносить context split или PO/MO/UNO правки маленькими партиями.
15. После каждого batch обновлять карты и verification report.

## Минимальный definition of done

Локализационный batch можно считать завершенным, если:

- все измененные строки есть в occurrence map;
- у всех измененных строк есть risk classification;
- нет измененных строк из запрещенных классов без отдельного решения;
- переводы внесены в правильный source class;
- активный контейнер реально содержит новые значения;
- rendered inventory после правки не показывает новые английские leftovers в затронутых местах;
- скриншоты подтверждают результат;
- RTL и ширина UI проверены для затронутых языков;
- есть отчет о false positives;
- есть rollback-план;
- update-log описывает, что было изменено и почему.

## Главное правило

ИИ-агент хорошо подходит для сбора карты, поиска пропусков, подготовки кандидатов и проверки регрессий. Но агент не должен сам "догонять все переводы" без контекста.

Правильная роль агента:

- собрать факты;
- показать все места употребления;
- разделить безопасное и опасное;
- предложить варианты;
- внести только утвержденные точечные правки;
- доказать результат рендером.

Правильная роль человека:

- утверждать терминологию;
- решать mixed-context случаи;
- выбирать архитектурный вариант разрезания ключей;
- принимать риск оставить английский термин;
- финально принимать визуальный результат;
- проверять качество перевода как носитель-предметник, а не только его наличие.
