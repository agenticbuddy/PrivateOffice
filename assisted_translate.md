# Ассистированная локализация Collabora Online

Этот документ описывает осторожный, инструментно-независимый рабочий процесс полной локализации проекта на базе Collabora Online с помощью ИИ-агента. Под "агентом" понимается любой помощник, который умеет читать проект, запускать команды и проверять результат.

Главная идея: локализация Collabora Online не сводится к заполнению пропущенных ключей в JSON. Сначала надо понять, где именно используется каждая английская строка, каким механизмом она переводится (ui-json / uno-json — клиентские каталоги, lo-core-mo — скомпилированные каталоги ядра, разные слои render-path), и есть ли у нее разные контексты (одна строка на разных экранах или в разных диалогах может требовать разного перевода), и только потом делать точечные правки.

Опасность в ложноположительном результате: если grep находит перевод, это не значит, что он рендерится на экране. Правка одного экрана может сломать другой.

Отдельный акцент — контроль ИИ-агента. Агент может уверенно сказать, что "проверил все языки" или "построил полную таблицу", но на практике часто делает выборку: несколько языков, пару вкладок, первые совпадения grep, часть каталогов. Поэтому результат принимается только тогда, когда агент оставил проверяемые артефакты: машинную таблицу (структурированный файл с данными, например CSV/TSV, а не текст в ответе), manifest сценариев (перечень проверенных экранов и шагов), список входных файлов с hash, скриншоты, DOM/text dumps (выгрузки текста со страницы или дерева DOM), команды воспроизведения (точные команды для повтора проверки) и числа, которые можно пересчитать независимо.

## Короткий вывод

Непереведённая строка в `ui-*.json` — это не задача «вписать перевод и закрыть». Один английский `msgid` может встречаться в разных местах интерфейса и требовать там разных переводов (коллизия контекстов). Часть подписей приходит вообще не из `ui-*.json`, а из UNO (внутренние команды/свойства LibreOffice) или из ядра LibreOffice-core, — источник нужно определять. Некоторые значения (форматы, техтермины, имена функций, бренды) правильнее оставить английскими. Лингвистически верный перевод может сломать UI: ширину кнопки, RTL-раскладку, tooltip, меню, диалог. Наконец, перевод может уже лежать в каталоге, но не появляться на экране, потому что строку рисует другой слой (иной уровень стека, отвечающий за отрисовку), — и стоковый языковой пакет LibreOffice нельзя подкладывать вслепую, так как Collabora отличается от него.

Почти вся локализация, тексты, размеры и дизайн элементов интерфейса достижимы без сборки ядра LibreOffice — они живут в слое Collabora Online (данные + CSS + немного кода). Сборка ядра нужна только для отрисовки самого документа.

Правильный путь:

1. Собрать все английские строки.
2. Собрать все места, где встречается каждая строка.
3. Разделить строки по источнику: `ui-json` (тексты интерфейса), `uno-json` (команды/свойства UNO), `lo-core-mo` (строки ядра LibreOffice), frontend i18n, generated/composed text (текст, собираемый из частей на лету), unresolved (источник не определён).
4. Построить карту контекстов и возможных коллизий.
5. Сначала переводить безопасные случаи.
6. Для рискованных случаев делать контекстное разделение (разные переводы одного `msgid` по месту употребления) либо править правильный источник: PO/MO (каталоги переводов LibreOffice) или UNO.
7. Проверять каждую правку рендером (открыть экран и посмотреть отрисовку), а не только grep-ом по файлам.

## Что именно сложно

В обычном приложении часто есть один слой локализации: код вызывает `t("Save")`, каталог хранит перевод, интерфейс показывает перевод.

В Collabora Online слоев больше:

| Слой | Что в нем живет | Как обычно чинить |
| --- | --- | --- |
| Frontend приложения | Обертка проекта вокруг Collabora: логин, файлы, профиль, модалки | `frontend/src/i18n/messages/*.json` |
| Collabora browser UI (веб-интерфейс: ribbon, меню) | Ribbon/notebookbar, часть меню, tooltips, панели, кнопки в web-коде | `ui-json` — данные в `editor/l10n/overrides/client/<lang>.json`, мёрджатся при сборке |
| UNO command labels (подписи команд LibreOffice) | Подписи команд LibreOffice, которые Collabora вызывает через `_UNO(...)` | `uno-json` — `l10n/uno/<locale>.json` или источник генерации этих каталогов |
| LibreOffice-core (ядро) | Диалоги, server/core UI, `.ui`, gettext, некоторые sidebar и dialog строки | `lo-core-mo` — `.po` (gettext-исходник) -> `.mo` (скомпилированный каталог), затем пересборка образа |
| Составные строки | Фразы, собранные из нескольких переводимых или непереводимых кусков | Переписать на цельную фразу с placeholders |
| Системные элементы | Браузерные меню, OS-native элементы, некоторые file picker/clipboard prompts | Обычно вне контроля приложения |

Правка не в том слое бесполезна: правка `lo-core-mo` не поможет строке из `ui-json`, и наоборот. Поэтому сначала надо определить источник.

Наличие строки в каталоге не означает, что текст берется именно оттуда. Пример: строка `Clipboard` есть в клиентском `ui-ru.json` с русским переводом, но в ленте показывается по-английски — потому что эту подпись рисует ядро, а клиентский каталог здесь не используется. Настоящий источник подтверждается только экспериментом: поменять строку в каталоге-кандидате, пересобрать, сбросить кэш и посмотреть, изменился ли экран.

Одна английская строка может лежать сразу в нескольких слоях; какой из них реально рисует текст, решает render path (путь отрисовки строки на экран; определяется экспериментом, см. раздел «Базовые термины»), а не факт наличия строки в каталоге. На практике большинство пропусков было в клиентских JSON, а не в ядре — там и ищи в первую очередь.

## Почему прямое добавление перевода опасно

Один и тот же английский ключ может требовать РАЗНОГО перевода в зависимости от места употребления, поэтому словарный перевод по тексту опасен.

Представим ключ `Illustrations`. Как название группы ribbon перевод `Иллюстрации` корректен. Но если есть составное сообщение:

```text
No + Illustrations
```

то пословный перевод даст:

```text
Нет Иллюстрации
```

а правильно:

```text
Нет иллюстраций
```

Английский (аналитический язык) терпит короткие словарные куски, собранные конкатенацией. Флективные языки — русский, украинский и многие другие языки со склонениями — от такой нарезки сообщений страдают сильнее: ломается грамматика (склонение, согласование).

Чек-лист рискованных многозначных слов:

| Английская строка | Почему рискованно |
| --- | --- |
| `Line` | Линия как фигура, строка текста, граница, линия диаграммы, команда меню |
| `Table` | Таблица как объект, команда вставки, стиль, вкладка, группа |
| `Field` | Поле формы, поле данных, поле сводной таблицы |
| `Function` | Функция Calc, команда вставки функции, раздел библиотеки функций |
| `Range` | Диапазон ячеек, область, именованный диапазон |
| `General` | Общий формат числа, общий раздел настроек, обычный режим |
| `Delete` | Удалить объект, удалить строку, удалить лист, удалить комментарий |

Вывод: привязывайте перевод к контексту и месту употребления (какой экран, диалог, роль строки), а не только к английскому тексту.

## Базовые термины

| Термин | Значение |
| --- | --- |
| `msgid` | Английская исходная строка или ключ, по которому ищется перевод |
| context | Окружение строки: приложение, вкладка, группа, команда, tooltip, соседние слова |
| collision | Один `msgid` используется в разных смыслах |
| composed fragment | Строка — часть фразы, а не самостоятельная фраза |
| source class | Слой (или слои) локализации, где живёт строка: `ui-json` (клиентский каталог), `uno-json` (каталог UNO-команд), `lo-core-mo` (скомпилированные PO/MO ядра LibreOffice), frontend i18n. Слоёв может быть несколько сразу; какой реально рисует текст, определяет render path |
| render path | Чем строка реально отрисована на экране: клиентский `_()`, серверный JSDialog из ядра или UNO (API вызова функций редактора). В отличие от source class, определяется экспериментом (рендером), а не наличием строки в каталоге |
| rendered proof | Доказательство через браузер: скриншот, DOM/text inventory (сбор фактического текста из DOM), состояние UI. Render — единственная истина |
| baseline | Зафиксированное состояние до правки (версия образа, каталоги, скриншоты, списки строк) как точка отсчёта |
| target locale | Язык, который надо проверять или допереводить |
| plural form | Форма слова, зависящая от числа. В русском 3 формы (1 файл / 2 файла / 5 файлов) |

## Общий рабочий процесс

Делите работу на этапы; не начинайте с массовой правки переводов.

Порядок этапов:

1. **Подготовка.** Определите зоны правки и инструменты: клиентский текст — `ui-json` (JSON-строки клиента), UNO-команды (внутренние команды движка LibreOffice) — `uno-json`, ядро — `lo-core-mo` (скомпилированные PO/MO — исходные/бинарные каталоги перевода). Уясните, что строка попадает на экран через свой render-path (слой, отвечающий за отрисовку), поэтому наличие строки в каталоге не гарантирует её показ.
2. **Правка переводов** в файлах-источниках соответствующей зоны.
3. **Проверка.** Ground truth — только render: сверяйте каждый элемент на живом экране (Playwright), а не по каталогу.

Критерий готовности — 100% переведённых строк, подтверждённых по render в каждой зоне.

## Инструменты инвентаризации (реализовано)

Инвентаризация термов для русского как референсного построена и воспроизводима. Команды пересборки и детали — `docs/localization/term-map.md`. Три инструмента:

- **Вселенная переводимого** (`build-l10n-catalog.py`) — все строки, что вообще можно перевести (client `ui/uno/locore` + core `.mo`), с текущим RU и классом источника (source class — где строка живёт: client ui/uno/locore, core `.mo`). Это знаменатель полноты.
- **Глубокий рендер-обход** (`editor-l10n-rendered.mjs`, Playwright) — проходит все UI-поверхности всех приложений, снимает видимый текст, геометрию и контекст (`app/surface/path/precondition` — приложение/поверхность/путь/предусловие открытия) и честно логирует `notReached` (недостижимые поверхности).
- **Карта термов** (`build-l10n-ru-terms.py`) — джойнит рендер со вселенной и классифицирует каждый терм: **simple** (одно место, один рендер, совпадает с каталогом), **complex** (омоним / render-path-mismatch — рендер расходится с каталогом / >1 значение каталога / multi-source — из нескольких источников), **composite** (склейка из фрагментов). Итог: `ru-terms.json` + CSV.

Классификация задаёт стратегию перевода.

### Как этим пользоваться при переводе

1. `simple` → переводишь точечно в нужном source class.
2. `collisions.csv` (complex) → решаешь по контексту (видны варианты рендера и места употребления), при необходимости разносишь ключи. `composites.csv` → переписываешь цельной фразой, не по кускам.
3. После каждого блока — регресс-тест `scripts/l10n-regression.sh`: подтверждает, что изменилось только намеченное, ничего не сломалось в других местах, и геометрия цела (нет перекрытий / обрезки / выхода за вьюпорт / нулевого размера). Ожидания ведутся в `expectations.json` поверх golden-снимка (эталонного снимка): терм переезжает из «должно остаться английским» в «переведено» — так отслеживается прогресс и ловятся регрессии.

Полнота считается по каталогу-знаменателю, но проверяется тем, что реально рендерится (рендер-обход + `notReached`), а не наличием строки в каталоге.

## Принцип доказательности

Единственный ground truth — то, что реально нарисовано на экране (скриншот). JSON-каталог, grep, `.mo` и факт вызова `_("...")` в коде доказывают лишь, что перевод *может* примениться, а не что применился здесь. Пример: `Clipboard` есть в `ui-ru.json`, но рисуется по-английски.

Поэтому источник и перевод каждой строки подтверждаются проходом **по каждому объекту** интерфейса (не по выборке): вкладка, группа, кнопка, меню, контекстное меню, диалог, тултип — со скриншотом на каждый блок. Проходов два: первый собирает, второй (желательно другой, более сильной моделью, которая перепроверяет собранное) ищет ошибки на скриншотах.

Этап заканчивается не словом «готово», а набором машиночитаемых артефактов, по которым другой человек/агент пересчитает результат. Ответ вида «проверил все строки, нашёл 20 проблем» не годится: неясны файлы, языки в scope, где остальные строки, полный это проход или выборка, воспроизводим ли он после обновления образа.

Результат перепроверяем, если по нему можно:

- открыть входной файл по hash и убедиться, что версия правильная;
- пересчитать число строк другим скриптом и сверить с числом в отчёте и в JSON/CSV;
- по любому `msgid` (идентификатор строки перевода) найти все его occurrence (места употребления в коде/интерфейсе);
- сопоставить любой скриншот с DOM/text dump;
- увидеть непокрытые места.

Кэш-ловушка: render-проверку ведут с cache-bust/hard-reload и в чистом профиле браузера. Часть ассетов (скрипт брендирования и рантайм-каталоги `ui-<loc>.json`) минует обычный cache-bust — ассеты кэшируются под неизменным хэшем, и старый экран легко принять за результат фикса. Поэтому дополнительно сверяют запросом мимо кэша.

### Анти-паттерны (не считать доказательством)

- общий текстовый ответ без raw-таблиц; ручные выводы без machine-readable артефакта;
- скриншоты только для одного языка при заявленных «всех языках»;
- grep-вывод без списка файлов и правил извлечения;
- отчёт «нашёл все» без общего количества сущностей; таблицу «top missing» вместо полной;
- перевод без всех occurrence; патч без обоснования выбранного source class (типа источника строки: `ui`/`uno`/`locore`/`.mo`);
- Playwright-проверку без manifest сценариев; отсутствие ошибок как доказательство полноты;
- наличие строки в каталоге (`ui/uno/locore-*.json` или `.mo`) как доказательство её отображения;
- render без cache-bust/hard-reload; один сброс кэша как гарантию свежести.

Каталоги различают, потому что строка приходит из разных источников: `ui` — клиентский UI (ленты, меню), `uno` — команды UNO (внутренние идентификаторы команд LibreOffice), `locore` — ядро LO, `.mo` — скомпилированные PO/MO-каталоги LibreOffice. Их нельзя смешивать: покрытие и патч у каждого свои.

### Минимальный набор проверяемых артефактов

| Этап | Что оставляет агент | Чем проверять |
| --- | --- | --- |
| Baseline | `baseline_manifest.json` с языками, приложениями, файлами, hash, версиями контейнеров | Пересчитать hash, сравнить языки с кодом проекта, проверить наличие файлов |
| Static extraction | `occurrences.json`, `coverage.csv`, `unresolved.csv`, команда запуска extractor | Перезапустить extractor, пересчитать regex/AST matches, проверить случайные строки |
| Rendered inventory | `scenario_manifest.json`, screenshots, DOM/text dumps, trace/log | Проверить наличие всех пар locale/app/state, открыть выборочные скриншоты |
| Collision map | `collisions.csv` со всеми `msgid`, risk class и occurrence ids | Проверить, что каждый `msgid` классифицирован ровно один раз |
| Translation candidates | `translation_candidates.csv` с source, context, confidence, reason | Проверить, что нет hard cases и unresolved |
| Patch | diff, список approved candidate ids, изменённые keys | Проверить, что diff меняет только разрешённые source classes |
| Verification | before/after inventory, screenshots, failures, false positives | Сравнить с patch list, проверить отсутствие новых конфликтов |

Пояснения к колонкам: coverage/collision map — таблицы покрытия и коллизий строк; risk class — класс риска замены (`single-use`, `same-context multi-use`); direct `_()` matches — прямые вызовы функции перевода в коде. Числа в артефактах (число уникальных `msgid`, UNO-команд, occurrence, произведения locale×app×state) не магические: каждое получено пересчётом из соответствующего входного файла и должно сходиться при перепроверке.

### Инварианты полноты

Для каждого этапа числа должны сходиться:

- число языков в отчёте = числу в `backend/app/services/locales.py`; для каждого языка есть строка в coverage table;
- каждый `msgid` из occurrence map имеет ≥1 occurrence, ссылающийся на существующий файл, offset, selector или screen state;
- каждый rendered English leftover связан с occurrence map или помечен `unresolved`; каждый `unresolved` имеет причину и следующий шаг;
- каждый safe candidate (кандидат на безопасную замену) имеет risk class `single-use` или `same-context multi-use`; hard case (неоднозначный случай — многократное употребление в разных контекстах) требует human decision либо явно остаётся нерешённым;
- каждый изменённый перевод имеет approved candidate id; каждый patch key после сборки реально присутствует в активном контейнере;
- каждый изменённый UI-state имеет before/after screenshot или text dump.

Если инварианты не сходятся, работа не завершена, даже при связном отчёте.

### Этап 1. Зафиксировать baseline

Baseline фиксирует, что именно локализуем:

- версия проекта; версия Docker-образа Collabora;
- список всех поддерживаемых языков;
- для каждого языка — цепочка кодов: код в приложении -> код в параметре `lang` -> реально загружаемый файл каталога;
- приложения: Writer, Calc, Impress, при необходимости Draw; режимы интерфейса, если дизайнов несколько;
- активные `bundle.js`, `l10n/ui-*.json`, `l10n/uno/*.json`, `l10n/locore/*.json`; LO resources `.po`/`.mo`/`.ui`, если доступны;
- тестовые документы Writer/Calc/Impress; UI-состояния: вкладки ribbon, меню, context menu, sidebar, dialogs, tooltips, формулы, таблицы, диаграммы, сводные таблицы, картинки, фигуры.

Цепочку код -> `lang` -> каталог проверяют для каждого языка отдельно. Пример: пользователь выбирает `ru` -> приложение передаёт код в `lang` (маппинг в `co_lang`: `ru` -> `ru-RU`) -> редактор нормализует регион и грузит `ui-ru.json`. Неверный код (или лишний регион) заставляет редактор молча не загрузить каталог или загрузить чужой — язык покажется непереведённым, хотя каталог на диске есть; иногда меняются и отдельные строки (подпись языка, placeholder вроде `{ru}`). Это частая незаметная причина нулевого покрытия.

Контроль, не основанный на доверии агенту:

- Запросить `baseline_manifest.json`, где каждая строка — конкретный вход: путь, источник, размер, hash, дата извлечения, контейнер/commit.
- Пересчитать список языков из кода и сравнить с manifest: если в проекте 22 языка, в manifest 22, а не «основные».
- Для каждого языка проверить, какие каталоги реально есть (`ui`, `uno`, `locore`, `.mo`); отсутствие каталога — явная строка в таблице, а не молчаливый пропуск.
- Scope приложений перечислен явно (Writer, Calc, Impress, shared, при необходимости Draw), excluded — тоже.
- Контейнерные файлы взяты из активного образа; доказательство — команда извлечения и hash файла.
- Для каждого языка проверить по network, какой файл каталога браузер реально запросил (не какой ожидался). Не запрошен ни один — скрытая причина нулевого покрытия.

Догон делают дельтой, не переписывая весь отчёт: достроить `baseline_manifest.json`, добавить все языки из `locales.py`, для каждого указать наличие ui-json/uno-json/locore-json/`.mo` и hash/размер, вывести отдельную таблицу missing catalog files, ничего не менять в коде. Критерий завершения — ровно N locale-строк, у каждой одинаковый набор колонок.

### Этап 2. Статическая инвентаризация строк

Агент читает код и каталоги, но рендер ещё не доказывает. Собрать: все `_()` строки; все `_UNO(...)` команды; строки frontend i18n; строки из `ui-*.json`, `uno/*.json`, `locore/*.json`; все PO/MO/UI ресурсы LibreOffice (PO/MO — текстовый/скомпилированный каталог переводов gettext); все места употребления каждой строки.

Ключевой результат — occurrence map: единый машиночитаемый реестр всех вхождений каждой английской строки. На его стабильные id потом ссылаются patch, collision map и verification, поэтому главный артефакт — `occurrences.json`/`occurrences.csv`, а не Markdown (Markdown допустим только как краткая сводка со ссылкой).

Минимальные поля occurrence map:

| Поле | Зачем нужно |
| --- | --- |
| `msgid` | Английская строка |
| `source_class` | Слой-каталог, где строка объявлена: `ui-json`, `uno-json`, `locore-json`, `lo-core-mo`, frontend i18n, unresolved (не найден источник). Может быть несколько значений сразу |
| `app` | Writer, Calc, Impress, shared |
| `file` | Файл или bundled segment |
| `line_or_offset` | Где найдено |
| `call_type` | `_()`, `_UNO(...)`, frontend `t()`, generated |
| `control_type` | tab, group, button, tooltip, menu, dialog, message |
| `neighbor_text` | Соседние слова или фраза |
| `command_id` | UNO command id (идентификатор команды движка) или control id, если есть |
| `is_fragment` | Строка — кусок фразы (`_("x")+var`, шаблоны); фрагменты нельзя переводить целиком, их помечают для ручной сборки |
| `render_path` | Слой, реально рисующий строку на экране (render-path): `client-_()`, `server-jsdialog-core`, `uno`. В отличие от `source_class` (где строка лежит), подтверждается только экспериментом |
| `notes` | Что проверить вручную |

`render_path` нельзя заполнять догадкой по каталогу. Его подтверждают экспериментом: поменять строку в предполагаемом источнике, пересобрать образ, сбросить кэш браузера, посмотреть, изменился ли экран. Изменился — источник верный; нет (как `Clipboard`: лежит в `ui-ru.json`, но рисуется ядром) — источник другой, правка этого каталога бесполезна.

Строку с экрана нельзя сравнивать с каталогом буквально. Перед сравнением нормализовать обе:

- убрать маркеры мнемоник (`~`, `_`);
- убрать хвостовое многоточие (`…`, `...`);
- убрать подсказку горячей клавиши в конце (`(Ctrl+S)`);
- убрать хвостовое двоеточие;
- строки с подстановками (`{1}`, `%1`) сравнивать как шаблон через регулярное выражение.

Без нормализации одна строка выглядит как разные, и появляется ложное «не переведено». Строки ядра в `.mo` сначала распаковать в текст (`msgunfmt`, пакет gettext).

Coverage-таблица (перевод есть в каталоге → строка считается переведённой) — верхняя оценка, гипотеза, а не факт: наличие в каталоге не значит, что именно он рисует текст. Процент подтверждает только рендер-проход (Этап 3). Обязательно взять выборку строк, числящихся переведёнными, и сверить с реальным экраном.

Требования к артефактам:

- `occurrences.json`/`csv` со ВСЕМИ occurrence (не sample), у каждого стабильный `occurrence_id`.
- Для каждого `msgid` — `occurrence_count` и ВСЕ места употребления (не первые несколько).
- Для каждого `source_class` — отдельный count; сумма по source_class сходится с общим итогом (`extraction_stats.json` с raw counts: `_()`, `_UNO(...)`, frontend keys, каталоговые keys) или явно показывает overlap.
- Для `unresolved` — не «unknown», а поле `why_unresolved`: нет command map, minified dynamic call, generated text, runtime-only, source вне бандла.
- `coverage.csv`: строка на каждый целевой язык, статусы missing/same/translated.
Как проверить полноту артефакта (не доверяя агенту):

- Другой командой пересчитать `_()` строки и `_UNO(...)` command ids, сверить с counts.
- Открыть вручную 20 случайных occurrence ids и проверить source location.
- Взять 20 случайных `msgid` с `occurrence_count > 1` и проверить, что перечислены ВСЕ места, а не первые.
- Проверить, что `coverage.csv` имеет строку на каждый язык со статусами missing/same/translated.

Признаки частичного/фальшивого выполнения: показан только top-20 missing; учтён только `ui-*.json` без `uno`/`locore`; заявлено «нашёл все» без extractor-команды; не отличают `msgid` от видимого текста; есть counts, но нет raw rows; у строки с несколькими occurrence показан один файл.

Отчет браузерной рендер-проверки (occurrence map = карта, где каждая строка UI встречена, с occurrence_id; rendered inventory = что реально нарисовано на экране) содержит:

1. Английские остатки.
2. Строки, которых нет в static map (статической карте строк из каталогов).
3. False positives: английские значения, которые вероятно должны остаться английскими.

Каждый английский остаток связывается с occurrence_id либо помечается `unresolved` — это критерий завершения.

**Контроль (не на доверии агенту).** До запуска — `scenario_manifest.json`: полный перечень пар `locale x app x state x action`. После — `run_manifest.json`: на каждый scenario id статус (`passed`/`failed`/`skipped`), время, пути к screenshot и text dump (текстовый снимок содержимого экрана). `skipped` недопустим без машинно-читаемой причины: missing fixture, feature unavailable, selector timeout, known out of scope.

- Каждый `state` — стабильное осмысленное имя (`calc.formulas.function-library-open`, не `screen5`).
- Для dropdown/tooltip нужен action log, доказывающий, что меню реально открывали.
- К каждому screenshot обязателен парный text dump: именно он дает автоматический diff и список английских остатков для сравнения с ручным отчетом.

**Независимые проверки.** Число строк в `run_manifest` == число строк в `scenario_manifest`. Для каждого `passed` есть и screenshot, и text dump. `failed`/`skipped` не исчезают из отчета. Выборочно открыть несколько screenshots и убедиться, что язык верный, а не reused image.

**Неочевидные признаки недоделки.** Нет `failed`/`skipped`, хотя реальный прогон почти всегда дает хотя бы timeout или out-of-scope. Tooltip/context menu заявлены проверенными, но нет action log.

**Как добить (delta-run).**

```text
Сделай delta-run:
1. Построй scenario_manifest.json для всех поддерживаемых языков, Writer/Calc/Impress и всех заявленных state.
2. Сравни существующие screenshots/text dumps с manifest.
3. Построй missing_scenarios.csv.
4. Прогони только missing_scenarios.
5. Обнови run_manifest.json, не удаляя failed/skipped.

Критерий завершения: run_manifest покрывает 100% scenario_manifest, каждый English leftover связан с occurrence_id или unresolved_id.
```

### Этап 4. Карта коллизий

После статической и рендер-инвентаризации определить, какие строки безопасны для перевода.

Классы риска (source class = класс источника строки):

| Класс | Что значит | Автоматический перевод |
| --- | --- | --- |
| `single-use` | Один `msgid`, один контекст | Обычно да, после проверки |
| `same-context multi-use` | Несколько мест, смысл один | Обычно да, если перевод везде подходит |
| `mixed-context` | Несколько мест с разным смыслом | Нет, нужен разнос или ручное решение |
| `composed-fragment` | Кусок составной фразы | Нет, переписать на цельную фразу |
| `source-mismatch` | Видно в UI, но источник не тот | Нет, сначала выяснить источник |
| `core-only` | Строка из LibreOffice-core | Через PO/MO (gettext-каталоги ядра), не через ui-json (плоский клиентский JSON) |
| `technical-name` | Формат, функция, бренд, имя команды | Часто оставить английским или сверить терминологию |
| `layout-risk` | Перевод не помещается или ломает layout/RTL/bidi | Только с визуальной проверкой |
| `render-path-mismatch` | Строка есть в ожидаемом каталоге, но рисуется из другого слоя (render-path = путь рендеринга), пример `Clipboard` | Нет: правка каталога не помогает, чинить реальный рендерящий слой |
| `plural-form` | Число-зависимые формы (1/2/5 в ru, 6 форм в ar) | Через gettext plural в ядре; плоский JSON их не выражает |

RTL-локали: отдельно проверять направление и раскладку (`dir`, положение меню, overflow) — учитывается в `layout-risk`.

**Команда агенту.**

```text
Разбей все английские msgid на классы риска на основе occurrence map и rendered inventory.
Учитывай все occurrence, разную грамматику и RTL, отдельно выдели составные фразы.

Результат:
1. Таблица collision map.
2. На каждый msgid: risk_class, occurrence_count, список контекстов, affected locales.
3. Safe candidates для автоперевода.
4. Hard cases для человека — с указанием, почему JSON-правка опасна.
```

**Контроль (не на доверии агенту).**

- Collision map покрывает ВСЕ `msgid` из occurrence map, не только проблемные; каждая строка ссылается на `occurrence_id` (иначе класс не доказан).
- `single-use` — ровно один occurrence id.
- `same-context multi-use` — несколько id и объяснение, почему контекст одинаков.
- `mixed-context` — перечислены разные context groups.
- `composed-fragment` — fragment pattern: какие куски соединяются и где видно.
- `technical-name` — почему английский допустим (формат файла, API-name, функция, бренд).

**Независимые проверки.** `set(msgid)` из `occurrences.json` == `set(msgid)` из `collisions.csv`, разница пустая. Строки с `occurrence_count > 1` не попали автоматом в `single-use`. Rendered leftovers тоже получили risk class, а не остались только в QA-отчете.

**Как добить.**

```text
Возьми полный set msgid из occurrences.json и текущий collisions.csv.
Построй missing_collision_rows.csv; на каждую строку добавь risk_class,
для occurrence_count > 1 добавь context_groups. Переводы не предлагай.

Критерий завершения: set(msgid) в occurrences.json полностью равен set(msgid) в collisions.csv.
```

### Этап 5. Подготовка переводов

Перевод готовить как контролируемую таблицу, не как свободный текст. На каждого кандидата: английский `msgid`; контекст; source class; текущий перевод по языкам; предлагаемый перевод; источник предложения (существующий LO-перевод, текущий каталог, терминбаза, ручной, ИИ); статус (`safe`/`needs human`/`rejected`/`accepted`); примечания по грамматике/длине/RTL.

Просить не «переведи все», а «предложи кандидаты, но не вноси».

**Команда агенту.**

```text
Подготовь кандидаты переводов, файлы не меняй.

- Используй ТОЛЬКО single-use и same-context multi-use.
- Не трогай mixed-context, composed-fragment, core-only и unresolved.
- Учитывай UI-контекст для каждого языка.
- При неуверенности ставь needs human, не угадывай.

Результат:
1. Таблица translation candidates.
2. На каждый msgid: контекст, source class, affected locales, proposed translation, confidence, reason.
3. Строки, которые нельзя переводить автоматически.
4. Термины, требующие единого глоссария.

Контроль полноты:
- Hard cases в таблицу не попадают.
- У каждого перевода есть причина.
- У каждого языка статус: proposed, existing, needs human, keep English.
```

### Этап 5 (завершение). Матрица safe candidates

Из отсеянных safe candidates строится таблица переводов. Контроль не строится на доверии агенту — проверяется результат, а не заявления.

**Формат.** `translation_candidates.csv` ссылается на `msgid`, `risk_class` (класс риска строки), `occurrence_id` (id места употребления), `source_class` (класс источника строки) и `locale`. Каждая строка имеет статус; пустой статус = работа не завершена.

**Провенанс.** Перевод без `source` и `reason` не принимается. `source` — один из: `existing-locore` (готовый перевод из LibreOffice-core), `existing-uno` (из UNO — команды/действия движка), `LibreOffice-po` (из .po-каталога LibreOffice), `glossary`, `human`, `ai-proposed`. У `ai-proposed` пониженное доверие, пока человек не утвердил; для непроверенных строк обязателен статус `needs human` с конкретным вопросом человеку, а не имитация уверенности.

**Разделение предложенного и внесённого.** Candidate table сама НЕ меняет файлы локализации — правки идут только после явного выбора человека.

**Заметки.** Для длинных переводов — layout note: риск overflow, нужен screenshot и shortened variant. Для RTL — note о порядке placeholders, пунктуации, направлении.

**Матрица как критерий полноты.** Строится матрица `msgid × locale`; каждая ячейка имеет статус `existing` / `proposed` / `needs human` / `keep English`. Пустых ячеек быть не должно. Ни одна hard-case строка (mixed-context / unresolved / коллизионная) не попадает в `proposed` без human approval.

**Независимые проверки batch:**

- risk_class всех строк — `single-use` или `same-context multi-use` (одноразовое / многоразовое в одном контексте), если задача только про safe candidates;
- в batch для `ui-json` (браузерный JSON-каталог клиентского UI) нет строк с source_class `lo-core-mo` (скомпилированный .mo core LibreOffice);
- каждый locale имеет статус по каждому candidate;
- glossary terms используются одинаково;
- 20 случайных переводов проверены по occurrence-контексту.

**Критерий завершения:** в матрице нет пустых ячеек, ни одна hard-case строка не попала в `proposed` без human approval.

### Этап 6. Разбор трудных случаев

Строки, отсеянные из safe-таблицы как hard cases (коллизии, составные фразы), обрабатываются отдельно: агент готовит decision package для человека. Ничего не меняет, только предлагает.

**Содержимое пакета:** `decision_id` (чтобы будущий patch ссылался на утверждённое решение, а не на свободный текст); где строка видна; все occurrences; причина коллизии; затронутые языки; ранжированные варианты с плюсами и минусами; что менять в коде или каталогах; verification plan (тесты + скриншоты).

**Требования к вариантам.** Дать 3–5 ранжированных вариантов (до 20 — только если простого пути нет), от самых реалистичных. Каждый вариант:

- перечисляет affected occurrence ids;
- при split key (разбиении одного ключа на несколько) показывает карту `old occurrence ids → new key ids`;
- при правке PO/MO/UNO (см. классы источников) обосновывает, почему клиентский `ui-json` — не тот источник (строки core/UNO не живут в браузерном JSON);
- при `keep English` объясняет, почему это приемлемо: термин, бренд, техническое имя, нет безопасного контекста;
- перечисляет новые риски: коллизии, fallback, layout, RTL, migration, rollback.

Рекомендуемый вариант указывается отдельно, но без реализации.

**Проверки полноты пакета:**

- все occurrences покрыты хотя бы одним вариантом;
- recommended option покрывает не только «лёгкую» часть occurrences;
- split keys не создают технических fallback-строк в английском UI;
- у каждого варианта есть verification plan;
- человек явно выбирает вариант до любых правок.

**Критерий завершения:** человек может выбрать вариант, не задавая уточняющих вопросов о том, где строка встречается и что сломается.

## Как делать точечные правки

Общий шаблон для команд агенту ниже одинаков (Задача / Контекст / Результат / Контроль полноты); в каждом разделе указаны только отличия. Общий принцип контроля: не доверять агенту на слово — каждый шаг подтверждается артефактом (approved-список, git diff, before/after каталог, occurrence map, отрендеренный сценарий). Термины: `occurrence` — конкретное место употребления строки в коде; `single-use` — одно occurrence; `same-context multi-use` — несколько occurrence с одним смыслом; `mixed-context` — одна строка в разных смыслах; `composed-fragment` — строка-кусок составной фразы; `same-as-English` — перевод намеренно совпадает с английским. Каталоги строк: `ui-json` — клиентский UI (кнопки, меню), `uno-json` — команды UNO, `lo-core-mo` / `locore-*.json` — ядро LibreOffice; принадлежность строки определяется по её источнику в коде.

### Безопасный direct `ui-json`

Править напрямую можно, если строка: из `ui-json`; `single-use` или `same-context multi-use`; перевод одинаково подходит всем occurrence; не составная фраза; не ломает ширину/RTL; есть план рендер-проверки.

Правки кладут как ДАННЫЕ в `editor/l10n/overrides/client/<lang>.json`; на шаге сборки deployed-образ (`editor/Dockerfile.online`) мёрджит их в активный `ui-<locale>.json` (`editor/Dockerfile` — только stopgap/fallback). Уже собранный файл в контейнере вручную не менять — иначе правка невоспроизводима при пересборке.

Утверждённый список safe candidates (`approved_ui_json_candidates.csv`) готовится до правки: в него отбираются строки, прошедшие критерии выше; агент работает только по нему.

Команда агенту:

```text
Ты исполнитель точечной localization-правки.

Задача: внести только safe direct ui-json переводы.

Контекст:
- Разрешены ТОЛЬКО msgid из approved_ui_json_candidates.csv.
- Не трогай mixed-context, composed-fragment, uno-json, lo-core-mo.
- Клади переводы как данные в editor/l10n/overrides/client/<lang>.json.

Результат:
1. Минимальная правка.
2. Список добавленных ключей по языкам.
3. Обновленный WAL/update-log.
4. Команды проверки.
```

Независимые проверки (не на доверии агенту):

- `set(ключей в diff)` минус `set(approved candidates)` = пусто; diff меняет только разрешённые файлы и ключи.
- После сборки извлечь активный `ui-<locale>.json` из контейнера и доказать, что ключ реально появился там, а не только в исходном патче (сравнить каталог before/after).
- Значение не перезаписало более свежий или иной существующий перевод в этом языке.
- `same-as-English` не считается ошибочно как «не переведено».
- UI не показывает технические ключи; для каждого ключа есть rendered scenario (id отрендеренного экрана как доказательство) или причина, почему рендер на этом batch невозможен.

Если патч шире scope:

```text
Патч шире утвержденного scope. Откати только свои изменения этого batch и пересобери patch.
Разрешено менять ТОЛЬКО keys из approved_ui_json_candidates.csv.
Выведи таблицу: approved key | changed file | old value | new value | rendered scenario id.
Критерий завершения: git diff содержит только разрешенные ключи; update-log описывает только этот batch.
```

### Контекстный разнос в коде

Один `msgid` в разных смыслах нельзя развести одним переводом. Программный контекст (`msgctxt`/`pgettext`) есть только в ядре (`.po`/`.mo`); клиентские каталоги — плоские словари «английская строка -> перевод» без техконтекста. Поэтому омоним на клиенте разводится только разными ключами — правкой вызова в коде.

Решения:

1. Если контекст на клиенте недоступен — добавить контекстный helper, который ищет скрытый контекстный ключ, но для английского fallback показывает нормальную английскую строку.
2. Составную строку заменить цельной фразой с placeholders.
3. Строку из UNO/core не разносить в browser JSON — чинить источник UNO/LibreOffice.

Английский fallback НЕ должен показывать технические ключи. Плохо: `_("Line|shape-tool")` — при отсутствии перевода пользователь увидит `Line|shape-tool`. Лучше: `translateContext("shape-tool", "Line")`, где fallback остаётся `Line`.

Команда агенту (отличия от общего шаблона):

```text
Ты разработчик контекстной локализации.

Задача: минимальный способ разделить конфликтующий msgid без поломки английского fallback.

Контекст:
- Один английский текст в нескольких разных смыслах; простая запись в ui-json небезопасна.
- Сохрани нормальный английский интерфейс при отсутствии перевода.

Результат:
1. Предложение API/helper или существующий механизм контекста.
2. Места замены вызовов и новые контекстные ключи.
3. Fallback behavior.
4. План миграции переводов, тесты и рендер-проверки.
```

Независимые проверки:

- До кодовой правки — таблица `split_plan.csv`: old msgid, old occurrence ids, new key, new context, fallback.
- После правки пересобрать occurrence map (соответствие occurrence -> ключ, строится обходом кода) и доказать, что каждый old occurrence либо распределён по новому ключу с ровно тем context, ради которого создан, либо явно documented keep-old; old msgid больше не используется в конфликтном месте.
- Английский UI без перевода показывает нормальный текст, а не `Line.shapeTool` или `shape-tool|Line`.
- Новые ключи не коллизируют с существующими ключами другого смысла.
- Placeholder order проверен для языков с другим порядком слов; rendered scenarios прогнаны для ВСЕХ старых мест, не только одного.

Если split неполный:

```text
Контекстный split неполный. Новые переводы не добавляй.
1. По split_plan.csv для каждого old occurrence id покажи, куда он попал.
2. Построй missing_or_unsplit_occurrences.csv и исправь только эти места.
3. Пересобери occurrence map.
Критерий завершения: все old occurrence ids имеют new key или documented keep-old reason.
```

### UNO и LibreOffice-core

Строки `_UNO(...)` и LibreOffice-core требуют особой осторожности; простая правка `ui-json` для них запрещена.

Правильный путь:

- найти исходный UNO command или LO resource, проверить наличие PO-контекста;
- править PO, а не бинарный `.mo`; собрать `.mo` через gettext toolchain;
- делать это на шаге сборки образа (Dockerfile), а не в работающем контейнере — иначе изменение пропадёт при пересоздании;
- `gettext` (`msgfmt`/`msgunfmt`) обычно не входит в образ — доустановить;
- убедиться, что Collabora реально использует новый ресурс, и проверить все команды с тем же английским текстом.

Команда агенту (отличия от общего шаблона):

```text
Ты специалист по LibreOffice/UNO локализации.

Задача: подготовить план исправления строки из uno-json или lo-core-mo.

Контекст:
- Простая правка ui-json ЗАПРЕЩЕНА.
- Найди правильный источник: UNO map, PO, MO, UI resource или generator.
- Не сломай другие команды с таким же английским текстом.

Результат:
1. Источник строки и способ правки (PO, не .mo).
2. Шаг сборки .mo и внедрения в образ.
3. Доказательство, что Collabora использует новый ресурс.
4. Проверка всех команд с тем же английским текстом.
```

Независимые проверки:

- Diff меняет PO-источник, а не бинарный `.mo`; сборка `.mo` выполняется на шаге образа.
- Активный ресурс в контейнере содержит новый перевод (сравнить before/after).
- Все команды с тем же английским текстом проверены на рендере и не сломаны.

## Правка UNO/core-строк: контроль и отказ

Прежде чем что-то менять, агент обязан показать полную **source chain** — цепочку от видимого текста до файла в собранном образе:

1. visible text и состояние экрана (где строка видна);
2. command/dialog id — команда UNO или диалог, где она используется;
3. исходный каталог: PO/UI-файл (**PO/MO** — gettext-каталоги перевода) или генератор;
4. build command для пересборки `.mo` или **uno-json** (генерируемый UNO-каталог строк);
5. active container path — путь в активном контейнере/образе;
6. все другие occurrences этого `msgid` в UNO/core;
7. rollback-план.

Нет цепочки — правки нет.

Требования к плану:

- Не редактировать `.mo` вручную как основной путь.
- Не править **ui-json** (клиентский каталог строк), если строка приходит из **core** (движок LibreOffice/диалоги). Определять это по source chain (шаги 2–3): если id — команда/диалог движка, источник — core, а не ui-json.
- Проверять `msgctxt` (контекст строки) в `.po`/`.mo`: один `msgid` может встречаться в нескольких контекстах — нельзя менять все разом без анализа коллизий.
- Проверять не файл `.mo`, а **рендер** — видимый UI: Collabora может брать строку из generated uno-json или другого кеша (render — единственный ground truth).
- Найти ВСЕ вхождения того же английского текста в UNO/core, не один экран.

Что человек проверяет независимо от агента (машинно-проверяемые артефакты):

- `msgunfmt` (gettext-инструмент) показывает ожидаемый `msgid`/`msgstr` в итоговом `.mo`.
- Дифф итогового `.mo` относительно **baseline** (исходной эталонной версии) меняется только в ожидаемом месте.
- Active container содержит новый `.mo` или regenerated uno-json.
- **Rendered inventory** (снимок фактически отрисованных строк) подтверждает изменение в нужном месте.
- **Collision map** (карта коллизий msgid по контекстам) обновлена, если PO-context разделяет смысл.

Критерий завершения: агент может объяснить, почему правка не затронет другой смысл той же английской строки.

Шаблон отказа, когда план по UNO/core неполный:

```text
План для UNO/core неполный. Не меняй файлы.
Дай source chain (пункты 1–7 выше) и объясни,
почему правка не затронет другой смысл той же английской строки.
```

## Как контролировать агента

Доверять надо не агенту, а артефактам (файлам-результатам: manifest, таблицы, counts, списки gap). Результат принимается, только если counts (числа) сходятся и присутствуют все ожидаемые строки.

### Контракт задачи

Задачу агенту дают как контракт с обязательными артефактами и жёстким scope. Сначала read-only анализ, потом правки маленькими партиями. Scope фиксируется до работы и не меняется без отдельного списка changes.

Каждая правка обязана:

- ссылаться на конкретный источник решения — approved candidate или decision id (утверждённый вариант перевода / запись решения);
- работать только в рамках safe candidates (проверенных вариантов) и разрешённого source_class (класса источника строки, напр. `ui-json` — строки клиентского UI из json; `uno-json` — строки UNO-команд, действий редактора; `lo-core-mo` — строки ядра LibreOffice из mo);
- не трогать рискованные risk_class: mixed-context, composed-fragment, core-only, unresolved (смешанный контекст, собранный из фрагментов, только-ядро, неразрешённый);
- иметь rendered proof (before/after — снимок реального экрана до/после) для каждого изменённого UI-сценария, на реальных языках и приложениях: ru/uk/ar/zh-CN, Writer/Calc/Impress.

Обязательные артефакты (файлы):

1. machine-readable manifest (`run_manifest.json`);
2. полная CSV/JSON таблица;
3. Markdown summary;
4. counts/invariants (инвариант — контрольное равенство, по которому counts сходятся);
5. список gaps/skips/unresolved;
6. команды воспроизведения (как повторить extraction — извлечение строк).

Каноничный шаблон задачи:

```text
Роль:
Ты не переводчик, а аудитор локализации. Твоя цель - доказуемо собрать полный artifact.

Scope:
Перечисли включённые языки, приложения, файлы, UI-состояния. Не расширяй и не сужай scope без отдельного списка changes.

Forbidden:
Не делай summary-only и sample. Не пропускай failed/skipped. Не вноси правки без ссылки на approved candidate.
Внеси переводы только для source_class=ui-json. Не трогай risk_class mixed-context, composed-fragment, core-only, unresolved.

Required artifacts:
1. machine-readable manifest;
2. full CSV/JSON table;
3. Markdown summary;
4. counts/invariants;
5. list of gaps/skips/unresolved;
6. reproduction commands.

Acceptance:
Работа принимается ТОЛЬКО если counts сходятся и все expected rows присутствуют.
Проверь: пересобери образ, открой Writer/Calc/Impress на ru/uk/ar/zh-CN, приложи rendered evidence.
Если любой шаг не получается или не успеваешь всё покрыть, остановись и выдай missing_work.csv, не угадывай.
```

### Как ловить частичное выполнение

Признаки неполного прохода: подсунута выборка вместо полного набора («основные языки», «ключевые экраны», «например», «top N» вместо полного списка); нет строк для failed/skipped; нет raw JSON/CSV; нет hash входных файлов; нет способа повторить extraction; нет связи summary с raw rows; нет ссылок от patch к approved decision/candidate; нет before/after evidence.

Не проси «проверь внимательнее» — получишь ещё одно summary. Дай delta-задачу на конкретный недостающий артефакт с явным критерием завершения.

### Как заставлять дорабатывать до полного результата

Механика доработки через множества:

1. Построить expected set (всё ожидаемое: языки, приложения, сценарии, `msgid`).
2. Построить actual set (что реально проверено и есть в таблице/скриншотах).
3. Построить gap set = `expected - actual`.
4. Прогнать ТОЛЬКО gap set — это не даёт снова делать выборочный общий проход.
5. Пересчитать invariant.

Критерий завершения: `missing_after = 0` либо у каждой оставшейся строки статус `blocked` с причиной.

Пример команды:

```text
Не делай повторный общий аудит.

1. Прочитай expected_scenarios.csv.
2. Прочитай run_manifest.json.
3. Построй missing_scenarios.csv как expected - actual.
4. Прогони только missing_scenarios.csv.
5. Обнови run_manifest.json.
6. Выведи counts: expected, before_actual, missing_before, added, missing_after.

Критерий завершения: missing_after = 0 или каждая оставшаяся строка имеет статус blocked с причиной.
```

### Чек-лист приёмки

Вывод принимается, только если выполнены все пункты; при отсутствии хотя бы одного — результат предварительный:

- есть полные expected set, actual set и gap set;
- есть machine-readable rows и воспроизводимая команда;
- counts сходятся;
- skipped/failed не скрыты;
- raw rows можно проверить выборочно;
- patch ссылается на approved ids;
- rendered proof покрывает изменённые states.

## Как проверять полноту

Одного скриншота или одной сверки мало: полнота доказывается несколькими независимыми сверками, где числа сходятся (инварианты). Иначе это пересказ, а не аудит.

Минимальный набор независимых сверок:

1. Static extraction (сколько строк найдено в коде).
2. Catalog coverage (сколько строк есть в каталогах по каждому языку).
3. Occurrence map (где каждая строка используется).
4. Rendered inventory (что реально видно в браузере).
5. Collision map (где один английский текст имеет разные смыслы).
6. Rendered regression + visual proof (что изменилось после патча, скриншоты до/после).
7. False positives (английские строки, которые должны остаться).

Обязательно: все 22 языка проходят одинаковый набор сценариев, а не выборка языков и вкладок. Дополнительно ищем строки, которые видны в рендере, но отсутствуют в статике, и наоборот; коллизии одного перевода на разные смыслы; составные фразы из отдельных слов; поломку RTL-языков; вылезшие из кнопок подписи; технические ключи вместо английского fallback; пропавшие иконки, тултипы и меню.

Полнота и качество — разные вещи. Сходящиеся числа доказывают, что перевод есть, виден и результат воспроизводим, но не доказывают его корректность. Корректность проверяется отдельным проходом сильной модели прямо по рендеру и финальной полировкой носителем-предметником (знает доменную терминологию).

### Проверки по типам работы

Термины: static extraction — извлечение строк из кода; occurrence map — где строка используется; collision map — карта одинаковых текстов с разным смыслом; catalog coverage — покрытие каталогов переводами; context split — разведение одного текста по контекстам; source_class — слой-источник строки (ui/uno/locore/.mo); risk_class — оценка риска перевода; source chain — цепочка «строка → её источник до правки». Артефакты (`*.json`, `*.csv`, coverage matrix, manifest) — файлы отчётов, которые генерирует агент; каждый требуется как доказательство.

| Работа | Что агент обычно недоделывает | Как проверить независимо | Как добивать |
| --- | --- | --- | --- |
| Baseline | Берет часть языков или не тот контейнер | Сравнить locale set с кодом, пересчитать hash извлеченных файлов | Требовать `baseline_manifest.json` и `missing_catalogs.csv` |
| Static extraction | Дает summary вместо полного occurrence map | Пересчитать `_()`, `_UNO(...)`, i18n keys другим скриптом | Требовать `occurrences.json`, `extraction_stats.json`, `unresolved.csv` |
| Catalog coverage | Проверяет только `ui-*.json` | Сравнить `ui`, `uno`, `locore`, `.mo` отдельно | Требовать coverage matrix по каждому source class |
| Rendered inventory | Проверяет один язык или одну вкладку | Сравнить `scenario_manifest` и `run_manifest` | Требовать `missing_scenarios.csv` и delta-run |
| Collision map | Классифицирует только очевидные проблемы | Сравнить set(msgid) occurrence map и collision map | Требовать полное покрытие всех `msgid` |
| Translation candidates | Переводит все подряд | Проверить risk_class и source_class для каждой candidate row | Удалить hard cases из batch и требовать статусы по каждой locale |
| Direct ui-json patch | Меняет больше ключей, чем утверждено | Сравнить diff keys с approved candidates | Сузить patch до approved set |
| Context split | Разделяет часть occurrence, оставляет конфликтные места | Сравнить old occurrence ids с new occurrence ids | Требовать `missing_or_unsplit_occurrences.csv` |
| UNO/core patch | Предлагает не тот слой исправления | Проверить source chain до `.po/.mo/uno json` и rendered UI | Требовать source-chain report до правки |
| Verification | Показывает только удачные скриншоты | Проверить failed/skipped и before/after text dump | Требовать полный verification manifest |
| Upgrade maintenance | Переносит старые патчи вслепую | Сравнить old/new occurrence map и changed contexts | Требовать release diff matrix |

Здесь ui/uno/locore/.mo — четыре слоя-источника строк: ui-json (клиентский интерфейс), uno-json (команды UNO — внутренние команды движка LibreOffice), lo-core (ядро LO) и его скомпилированные `.mo` (бинарные каталоги из `.po`).

### Пример независимых инвариантов

Агент обязан показать, КАК число пересчитано другим способом, иначе отчёт не принимается. Пусть статический отчет утверждает:

```text
locales = 22
direct_ui_msgid = 436
referenced_uno_labels = 300
global_uno_labels = 813
```

Тогда должны существовать независимые проверки этих чисел:

```text
count(locales_from_project) == 22
count(unique direct _() msgid in selected code segments) == 436
count(unique resolved UNO labels from referenced commands) == 300
count(unique labels in global UNO map) == 813
```

### Как принимать частично заблокированную работу

Иногда полный проход невозможен: Playwright занят, fixture не открывается, источник minified, `.po` отсутствует, команда создается динамически. Blocked-состояние легитимно, но должно быть явным. Для каждого blocked item нужны поля:

| Поле | Что писать |
| --- | --- |
| `blocked_id` | Стабильный id |
| `scope_item` | Что именно не покрыто |
| `reason` | Почему не покрыто |
| `attempted` | Что уже пробовали |
| `needed` | Что нужно, чтобы закрыть |
| `risk` | Что может быть неверным из-за блока |
| `owner` | Агент, человек, внешний инструмент |
| `next_check` | Как вернуться к проверке |

Фраза "не удалось проверить" без этих полей не считается отчетом.

## Как вести итоговые артефакты

Кроме правок нужны документы, фиксирующие покрытие и решения.

Рекомендуемые файлы (baseline — стартовый снимок состояния; occurrence — каждое вхождение строки интерфейса; render-path/«слой» — где строка чинится: клиентский JSON или ядро `.mo`):

| Файл | Назначение |
| --- | --- |
| `localization_baseline.md` | Версия образа, список языков, покрытые приложения |
| `localization_occurrences.json` | Occurrence map: машиночитаемая карта всех вхождений с контекстами |
| `localization_coverage.md` | Сводка translated/missing/same по языкам |
| `localization_collisions.md` | Все контекстные риски |
| `translation_candidates.md` | Кандидаты переводов и их статусы |
| `translation_decisions.md` | Человеческие решения по hard cases |
| `translation_verification.md` | Итоговые проверки и скриншоты |
| `translation_glossary.md` | Термины, которые должны переводиться единообразно |

Стартовый baseline-артефакт (первый снимок инвентаря) в проекте уже есть:

```text
docs/COLLABORA_LOCALIZATION_STATIC_INVENTORY.md
```

Это не финал. Следующий шаг — occurrence map с контекстами и rendered inventory (перечень фактически отрисованных строк интерфейса).

По итогам прохода формируются два артефакта:

- `.qa/l10n/REPORT.md` — отчёт о покрытии: сколько видимых строк, сколько переведено по каждому языку, и разбивка пропусков на «чинится в клиентском JSON» и «только через ядро `.mo`».
- `.qa/l10n/visible-coverage.csv` — машинная матрица: по строке на каждую строку интерфейса, со статусом по каждому языку.

Вместе они дают проверяемую опору для решения, что и каким слоем чинить.

## Стратегия работы с исходниками Collabora

Главный принцип: **из исходников собираем только Collabora Online** — сервер coolwsd (backend-демон редактора) и браузерный bundle (собранный JS/CSS клиента). **Ядро LibreOffice/Collabora Office из исходников НЕ собираем** — берём его закреплённым готовым бинарём (engine-assets — предсобранные файлы движка LO). Строки ядра при этом правим как данные, а не пересборкой.

Причина: при глубокой локализации всё, что мы меняем в UI (тексты, размеры, дизайн), живёт в слое Online, а не в ядре; там же — склейки строк в JS и контекстный API перевода.

Быстрый путь, от которого мы уходим:

```text
FROM collabora/code:latest
```

дальше Dockerfile правит уже собранные `bundle.js`, `bundle.css`, `cool.html`, `l10n/ui-*.json` через `sed`/`cp`. Допустимо как временный костыль (брендовая мелочь, hotfix), но не как основа локализации, потому что:

- `latest` недетерминирован — сегодня и завтра это разные байты;
- minified `bundle.js` не даёт контекстного разделения строк (одинаковый текст с разным смыслом нельзя развести);
- `sed`-патч по собранному файлу ломается при любом upstream-изменении;
- нельзя нормально добавить контекстный API локализации;
- после обновления не отследить, какой исходный файл породил строку.

Версию фиксируем жёстко и **парой**: commit/тег Collabora Online И версия engine-assets ядра — обе в `upstream.json`, и они должны совпадать. Причина: строки (`msgid` — исходный текст, `msgctxt` — контекст-дискриминатор в gettext) версионно-зависимы. Поэтому обновление версии — отдельное контролируемое событие с полной приёмкой переводов, а не «подтянул latest». При сборке из исходников security-фиксы апстрима бэкпортишь сам — ещё одна причина держать обновление под контролем.

Ступени ниже — это лестница ухода от готового образа; конечная точка одна — сборка Online из закреплённых исходников. Если есть возможность, можно сразу переходить на целевой уровень.

### Ступень 1 (временная): убрать недетерминированность готового образа

Пока сборка из исходников не поднята:

1. Не использовать `collabora/code:latest`.
2. Закрепить образ по digest или tag.
3. Записать pinned image в baseline (зафиксированный эталон).
4. Хешировать активные артефакты: `bundle.js`, `ui-*`, `uno/*`, `locore/*`, `.mo`.
5. Любой отчёт строить только против этого baseline.

```text
collabora/code@sha256:<digest>
```

(шаблон команды агенту-release-инженеру для проверки pinning).

### Ступень 2 (промежуточная): патчи как единицы, а не длинные `sed`-цепочки

Пока образ ещё готовый, изменения держим не как `sed`-цепочки, а как описанные **patch units** — атомарные патчи с метаданными.

```text
editor/
  Dockerfile
  patches/
    collabora-online/
      0001-contextualize-line-label.patch
    libreoffice-core/
      0001-ru-pivot-field-translation.patch
  scripts/
    extract-active-collabora-assets.sh
    build-l10n-occurrence-map.py
  manifests/
    upstream.json
    patchset.json
```

Требования к патчу как unit совпадают с доказательством слоя (см. «Какой слой чинить»): что меняет, почему это правильный слой, к какому upstream digest относится, какие `occurrence ids` (идентификаторы конкретных вхождений строки) закрывает, какой rendered scenario подтверждает результат, как откатить.

### Целевой уровень: сборка из закреплённых исходников

Рабочая основа проекта. Из исходников собираем **Collabora Online**, держим не копию, а fork или submodule закреплённого commit. **Ядро LO НЕ собираем** — закреплённый готовый бинарь (engine-assets), его `.mo` (скомпилированные gettext-каталоги) правим как данные из наших `.po` (текстовые каталоги переводов).

Что какой слой позволяет менять **без сборки ядра**:

- **Тексты** — данные (клиентские `ui/uno/locore-<lang>.json` + `.mo` ядра из наших `.po`) плюс немного кода в Online для расклейки склеенных фраз;
- **Размеры** — CSS (часть — JS);
- **Дизайн** (цвета, формы, отступы, иконки) — CSS + замена SVG; так уже сделана glass-тема.

```text
third_party/
  collabora-online/       # submodule/fork, pinned commit (СОБИРАЕМ из исходников)
editor/
  engine/                 # engine-assets ядра LO (готовый бинарь, НЕ исходники)
  l10n/overrides/
    core/<lang>.po        # переводы строк ядра, с msgctxt → компилируются в .mo
    client/<lang>.json    # оверлеи клиентских строк
  patches/                # ТОЛЬКО код Online (контекстный API, развод склеек)
  build/                  # детерминированная сборка Online + наложение данных
  manifests/
    upstream.json
    patchset.json
```

Главное разделение — **переводы это ДАННЫЕ, патчи это только КОД**. Переводы (`.po` с `msgctxt` + JSON-оверлеи) детерминированно компилируются в `.mo` и каталоги, легко ревьюятся и масштабируются на 30–40 языков. Патчи — только код (новый контекстный API, переписывание склеенных фраз на цельные с placeholders). В одном изменении их не смешивают.

Fork/submodule лучше копии, потому что: сохраняется upstream history (rebase/merge); видно отличия от upstream; проще проверить применимость к новой версии; меньше риск править generated-вывод вместо исходника.

Куда отдавать наружу — каналы не путать: **переводы — в Weblate** (площадка переводов LibreOffice/Collabora); **изменения кода** (контекстный API, фиксы склеек) — обычным PR в Collabora Online или LibreOffice.

### Какой слой чинить (ядро или клиент)

Часть интерфейса приходит не из Online, а из LibreOffice-core (`.ui`-файлы, gettext `.po/.mo`, метаданные UNO-команд — внутренних команд движка LO, диалоги, sidebar). Слой строки — «клиент Online vs LO-core» — выбирается **доказательством, а не на глаз** (это render-path атрибуция — определение, каким путём строка попадает на экран). Перед каждой правкой докажи:

- строка реально из LO-core, а не из клиентского `ui-json`/`uno-json` (каталоги клиентских строк ленты/меню и UNO-команд);
- есть конкретный PO/UI/исходный файл;
- есть путь сборки до `.mo` или generated-ресурса;
- есть rendered scenario, подтверждающий результат на экране.

Клиентские строки правим в наших JSON-оверлеях, строки ядра — в наших `.po` с `msgctxt`.

### Когда всё же собирать ядро LO из исходников

Почти никогда. Только если надо менять **отрисовку самого документа** (как ядро рисует сетку Calc, слайд, текст в ячейках) или **C++-поведение ядра**. Это содержимое документа, а не элементы UI — для текстов, размеров и дизайна не нужно.

Это самый тяжёлый путь: часы даже на быстрой машине, образ до ~30 ГБ. Даже официальная сборка «CODE for Web» ядро НЕ пересобирает — подкладывает готовые engine-assets. Полная сборка ядра — необязательная эскалация, не наш дефолт.

### Что не надо делать

- копировать весь Collabora source без upstream-связи;
- жить на `latest`;
- редактировать minified `bundle.js` как основной способ правок;
- массово импортировать PO/JSON без occurrence/collision map (карты вхождений и конфликтов строк);
- смешивать app-переводы, browser-UI и LO-core в одном patch batch;
- держать переводы как код-патчи вместо файлов данных (`.po`/JSON-оверлеи);
- катить обновление версии в прод без полной приёмки переводов.

### Контроль полноты для source-стратегии

Source-стратегия = сборка editor из закреплённых исходников, а не правка готового образа. Для неё нужны отдельные артефакты:

| Артефакт | Что доказывает |
| --- | --- |
| `upstream.json` | Какая версия Collabora и LibreOffice используется |
| `patchset.json` | Какие патчи применяются, к чему относятся, какие occurrence (места вхождения строки) закрывают |
| `source-build.md` | Как собрать editor image из source или серии патчей |
| `source-diff-report.md` | Чем fork отличается от upstream |
| `generated-vs-source-map.json` | Как generated output (собранные/скомпилированные файлы, а не правятся вручную) связан с исходными файлами |
| `release-diff-matrix.csv` | Что изменилось при обновлении upstream |

Инварианты:

- editor image не должен быть `latest`;
- Online собирается из закреплённых исходников; ядро LO — закреплённый готовый бинарь/engine-assets; версия Online и версия LO-core зафиксированы в `upstream.json` и **совпадают** (согласованы между собой);
- переводы хранятся как данные (PO/MO — бинарные каталоги перевода, JSON-оверлеи), а не как код-патчи; код-патчи допустимы только для кода;
- каждый код-патч имеет upstream base commit, owner и reason; патч без привязки к commit или image digest не принимается;
- каждая запись перевода и каждый код-патч ссылается на occurrence ids; каждый source-level split обновляет occurrence map (трассируемость строк);
- generated output не редактируется, если доступен source-level путь;
- каждое обновление версии проходит полную приёмку до промоушена в прод и пересчитывает release diff matrix.

Команда агенту:

```text
Ты архитектор source-level стратегии для Collabora localization.

Задача: предложить, как перейти от патчей по готовому образу к воспроизводимой source-level схеме.

Результат:
1. Текущий способ сборки editor.
2. Риски текущего способа.
3. Минимальный pinning plan (фиксация версий).
4. Patch-series plan для текущего repo.
5. Fork/submodule plan для Collabora Online.
6. Отдельное решение, нужен ли LibreOffice-core source.
7. Список новых artifacts.
8. Migration plan по шагам.

Контроль полноты:
- Указан текущий image/tag/digest.
- Явно разделено, какие правки остаются image-level, а какие становятся source-level.
- Есть rollback plan.
- Есть стратегия обновления upstream.
- Есть способ проверить, что патчи применяются к правильной версии.
```

## Как поддерживать после обновлений Collabora

При обновлении Collabora нельзя вслепую переносить старый патч: обновление версии — это контролируемое событие с полной приёмкой, а не перенос одного патча.

Collabora — это не стоковый LibreOffice: у неё свои строки и свой нотбукбар. Поэтому любой импорт чужого перевода или языкового пакета сверяется по версии и контекстам.

Порядок такой:

1. Поднять новую закреплённую версию **сначала в непрод-окружении**, не трогая прод: новый commit Collabora Online + **совпадающую** версию готового ядра/engine-assets (готовые бинарники движка).
2. **Пересобрать Online из исходников** новой версии; ядро при этом берётся готовым бинарём/engine-assets и не пересобирается.
3. Прогнать **полную приёмку, а не smoke**:
   - пересобрать occurrence map (карту вхождений строк) и покрытие;
   - прогнать render-ground-truth (истина по факту отрисовки — скриншот по каждому объекту) по **всем** целевым языкам;
   - сделать drift-diff (сравнение расхождений строк) всего интерфейса против предыдущей версии.
4. Разобрать, что сломало обновление: переехавшие или переименованные `msgid`, изменившийся `msgctxt`, новые английские строки, удалённые строки, вернувшиеся склейки нескольких строк в одну.
5. Пере-применить (rebase) наши данные переводов и код-патчи на новую версию и починить поломки.
6. **Только после прохождения полной приёмки** — промоушен в прод.

Эта приёмка — обязательный CI-гейт до промоушена: те же проверки (покрытие, render-ground-truth, drift-diff) гоняются на каждом подъёме версии. Критерий принятия — скриншот по каждому объекту плюс второй проход-проверка (повторный контрольный проход, см. «Принцип доказательности»).

Команда агенту:

```text
Ты release-аудитор локализации после обновления Collabora.

Задача: сравнить старый и новый baseline и сказать, какие локализационные патчи безопасно переносить.

Контекст:
- Старые патчи могли быть привязаны к старым source strings.
- Новые PO/JSON (файлы переводов) могут иметь другие контексты.
- Нельзя массово переносить без проверки.

Результат:
1. Таблица unchanged keys.
2. Таблица removed keys.
3. Таблица new keys.
4. Таблица changed occurrences.
5. Список патчей и записей перевода, которые можно оставить.
6. Список патчей и записей перевода, которые надо пересмотреть.
7. План полной приёмки (не smoke): render-ground-truth по всем языкам + drift-diff против прошлой версии.

Контроль полноты:
- Каждый старый patch key должен быть классифицирован (unchanged/removed/new/changed).
- Каждый новый английский leftover (оставшаяся непереведённая строка) должен быть связан с source class (исходным классом строки).
- Если контекст изменился, патч/перевод должен быть помечен needs review.
```

## Масштабирование на много языков

Когда языков не 4, а 30-40, работу не повторяют для каждого. Она делится на два уровня:

- Поверхность снимается один раз. Occurrence map (список всех мест, где встречается каждая строка) не зависит от языка - его строят однократно.
- Покрытие по языку считается данными: строки сверяют с каталогами языка и его `.mo` (скомпилированный каталог перевода). Это чистый расчет по файлам, без браузера, масштабируется на любое число языков.

Данными не проверить, как перевод реально выглядит на экране: перенос строк, выход за кнопку, RTL-раскладка, контекстные коллизии (один и тот же ключ в разных местах требует разного перевода). Это видно только в рендере, и проверять надо по каждому языку. Браузер - узкое место (один на инстанс). Решение - распараллелить: несколько ВМ, по одному языку на машину, каждая со своим инстансом редактора и браузера; скрин-проходы идут параллельно.

## Практический порядок внедрения

Три фазы: инвентаризация и картирование, пакетная обработка safe-строк, ручная работа над hard cases.

1. Закончить occurrence map (карту всех вхождений переводимого текста) через `scripts/build-l10n-catalog.py` — автоматизировано, см. «Инструменты инвентаризации».
2. Сделать rendered inventory (что реально отрисовано на экране) через Playwright, `e2e/scripts/editor-l10n-rendered.mjs` — автоматизировано.
3. Построить collision map (карту конфликтующих термов) через `scripts/build-l10n-ru-terms.py` — автоматизировано.
4. Выделить safe candidates — строки с единственным непротиворечивым переводом (по occurrence и collision map); остальные идут в hard cases.
5. Согласовать глоссарий для частых терминов.
6. Внести маленький batch (партию) safe-строк в `ui-json` (клиентский JSON-каталог интерфейса).
7. Пересобрать образ.
8. Проверить активные каталоги — что в сборку попали именно нужные версии каталогов.
9. Прогнать Playwright smoke (беглая проверка) по всем языкам.
10. Прогнать полный Playwright по ключевым языкам: ru, uk, ar, zh-CN, плюс несколько европейских и RTL.
11. Исправить проблемы ширины, RTL и составных фраз.
12. Для hard cases решение выбирает человек.
13. Вносить правки context split (развод одинаковой строки по контекстам) или PO/MO/UNO (текстовые каталоги LibreOffice и UNO-строки) маленькими партиями.
14. После каждого batch обновлять карты (occurrence, collision) и verification report (отчёт о проверке).

## Минимальный definition of done

Локализационный batch завершён, если:

- все изменённые строки есть в occurrence map (реестре затронутых вхождений) — учёт полный;
- у каждой изменённой строки есть risk classification (класс риска); строки из запрещённых классов не трогаются без отдельного решения;
- перевод внесён в правильный source class (исходный контейнер строки) и реально присутствует в активном контейнере на рантайме — то есть подхватился, а не только записан;
- rendered inventory (инвентарь по фактическому render — он и есть ground truth) после правки не показывает новых английских остатков в затронутых местах;
- регресс-тест (`scripts/l10n-regression.sh`) зелёный — без падений: нет text-changes вне намеченного блока и нет новых layout-дефектов (перекрытия / обрезка / выход за вьюпорт); подтверждено скриншотами;
- RTL и ширина UI проверены для затронутых языков;
- есть отчёт о false positives (ложных срабатываниях);
- есть rollback-план (откат изменений);
- update-log описывает, что изменено и почему.

## Роль агента и человека

ИИ-агент готовит и проверяет перевод, но не переводит автономно: он не «догоняет переводы» без контекста. Утверждает термины и принимает результат человек.

Роль агента:

- собрать факты и показать все места употребления строк;
- разделить безопасное и опасное;
- предложить варианты;
- внести только утверждённые точечные правки;
- доказать результат рендером (реальным экраном, а не наличием строки в каталоге).

Роль человека:

- утверждать терминологию;
- решать mixed-context (одна строка в разных смыслах);
- выбирать вариант разрезания ключей (разделения одного ключа на разные по контексту);
- принимать риск оставить английский термин;
- финально принимать визуальный результат;
- проверять качество перевода как носитель-предметник — наличие строки не равно её качеству.
